# Phase 4.1 — Portal grading → Celery migration design

> **Status:** FINAL. Codex Gate 1 GREEN pending (after this spec is written; 2 prior rounds HOLD→resolved). Gemini reviewed revised design with 9 refinements; all incorporated.

**Date:** 2026-04-16
**Scope parent:** Phase 4 of `notes/infrastructure/2026-04-12-codebase-improvement-roadmap.md` (Architecture Evolution), deliverable #1: *"Task queue (Celery + Redis) replacing daemon-thread grading."*
**Scope of this spec:** Phase 4.1 — migrate **join-code portal grading** path only. Class-based student-account path deferred to 4.1b. Batch file grading deferred to 4.1c.

---

## Goal

Replace the daemon-thread portal grading pipeline with a durable Celery + Redis task queue running on a separate Railway worker dyno. Deliver three concrete wins:

1. **Durability** — current threads die on Railway restart; student submissions stuck as `grading_deferred`. Celery broker re-delivers on worker restart.
2. **Bounded concurrency** — 4-8 worker slots vs. today's unbounded `threading.Thread` spawn under classroom-sized load.
3. **Process isolation** — grading workload no longer competes with HTTP for gunicorn worker slots.

Phase 4.1 is also the foundation for subsequent Phase 4 sub-items (batch grading, future async work) that need Celery infrastructure in place.

## Non-goals

- **No class-based path migration.** `student_account_routes.py` keeps using `run_portal_grading_thread` — deferred to Phase 4.1b.
- **No batch grading migration.** `backend/grading/thread.py` / `pipeline.py` keep running on threads — deferred to Phase 4.1c.
- **No route-level double-submit dedup.** Pre-existing bug (same-student double-click creates two submissions rows via `student_portal_routes.py:781-785`). Flagged as follow-up.
- **No Flower/Celery monitoring UI.** Rely on Sentry + BetterStack for now.
- **No dead letter queue.** Rely on Sentry captures + manual intervention for `failed` submissions.
- **No Celery Beat.** No scheduled tasks in scope.
- **No Redis replacement for flask-limiter/flask-session.** Celery uses DB 1 (or whatever `CELERY_BROKER_URL` points to); web-side Redis unchanged.

## Hard constraints

- **Clever/ClassLink/OneRoster compliance** — NO modifications to `clever_routes.py`, `classlink_routes.py`, `roster_sync.py`, `oneroster_gradebook.py`, `student_account_routes.py`, or any SIS-adjacent file. SIS contract suite 180/180 green on every PR.
- **Coverage floor 32** holds (CI-enforced).
- **Boot check** ≥ 250 routes on every PR (`tests/test_app_boot.py`).
- **PRs must merge sequentially** — each PR's infra is referenced by the next.
- **Feature-flag gated rollout** — `CELERY_PORTAL_GRADING=1` enables Celery enqueue; default off means legacy thread path.

## Design

### Stack (roadmap-fixed)

- Celery + Redis
- Broker: Redis at `CELERY_BROKER_URL` (explicit env, fail-fast if unset). Recommended default: `${REDIS_URL}/1` when Railway Redis supports multiple DBs.
- Result backend: none. Supabase submission row is the source of truth.
- Serialization: JSON (Celery default).
- Worker process: new Railway service from same repo, shares `REDIS_URL` + `SUPABASE_*` envs.
- Worker pool (MVP): **prefork with `--concurrency=8`**. Rationale: Codex flagged gevent compatibility risk from `ThreadPoolExecutor` + `threading.Lock` usage in `assignment_grader.py`. Gevent (`--pool=gevent --concurrency=20`) deferred as flag-flip after staging load-test.

### Task design

```python
from celery import shared_task

@celery_app.task(
    name='grading.portal_submission',
    bind=True,
    acks_late=True,
    autoretry_for=(TransientError, TimeoutError),
    retry_backoff=True,
    retry_backoff_max=300,
    max_retries=3,
    time_limit=900,       # 15 min hard kill
    soft_time_limit=840,  # 14 min soft warning
)
def grade_portal_submission(
    self,
    submission_id: str,
    teacher_id: str,
    supabase_table: str,  # 'submissions' (4.1) or 'student_submissions' (4.1b future)
    *,
    district_id: str | None = None,
    user_id: str | None = None,
) -> None:
    """Run portal grading pipeline for one submission.

    Exactly-once semantics via row-level task_id + started_at dedup:
    - If submission.status == 'grading_in_progress' AND task_id != self.request.id
      AND started_at < 15 min ago → skip (another live worker has it).
    - Else → claim (update row with task_id + timestamp) and proceed.

    Handles:
    - Celery redelivery (same task_id → proceeds, idempotent re-grade of same submission)
    - Stale crash reclaim (timeout-based takeover by new task_id)
    - Teacher result duplication (save_results is upsert-by-submission_id, not append)
    """
    import sentry_sdk
    if user_id:
        sentry_sdk.set_user({"id": user_id})
    if district_id:
        sentry_sdk.set_tag("district", district_id)

    grade_portal_submission_sync(
        submission_id=submission_id,
        teacher_id=teacher_id,
        supabase_table=supabase_table,
        district_id=district_id,
        user_id=user_id,
        task_id=self.request.id,  # enables row-level dedup
    )
```

### Dedup mechanism

**Row-level dedup on the submission row** (NOT SETNX, which has stall risk if worker crashes mid-grade — lock holds for TTL before redelivered task can proceed):

New columns on `submissions` (and `student_submissions` for 4.1b):
- `grading_task_id` (text, nullable)
- `grading_started_at` (timestamptz, nullable)

Task entry logic:
```python
current = fetch_submission(submission_id)
if current['status'] == 'grading_in_progress':
    if current['grading_task_id'] == task_id:
        # Same task retrying — proceed (idempotent regrade)
        pass
    elif current['grading_started_at'] > now - 15min:
        # Another live worker owns it — skip
        return
    # else: stale claim, reclaim (fall through to update)
update_submission_claim(submission_id, task_id, started_at=now, status='grading_in_progress')
# ... grade body ...
# On success:
update_submission(submission_id, status='graded', ...fields...)
# On exception: Celery retries; status stays 'grading_in_progress' with same task_id
```

**Property:** Any path (initial try, retry, crash redelivery) converges on exactly one `status='graded'` state and one set of teacher Results.

### Teacher result idempotency

**Current state (verified):**
- `backend/services/portal_grading.py:565-567` appends to teacher storage (join-code path)
- `backend/grading/state.py:94-101` overwrites full list (batch path, unaffected by this PR)
- `build_result_record()` at `backend/services/portal_grading.py:201-224` does NOT include `submission_id` in stored records today — this is the load-bearing gap Codex flagged

**Required changes in PR2:**

1. **Add `submission_id` to stored record.** `build_result_record()` gains a `submission_id` field in its output dict. Existing Results/Analytics readers must tolerate the extra field (they ignore unknown keys — verify on a sample teacher). Any consumer that validates keys strictly gets an explicit tolerance update.

2. **Overwrite-by-submission-id in `save_results()`:**
```python
def save_results(teacher_id, result_record):
    """Idempotent by submission_id (added to record shape in PR2).
    Removes any existing record with the same submission_id, then appends fresh.
    Safe for Celery retries — multiple deliveries of the same task_id
    converge on exactly one stored record per submission_id.
    """
    existing = load_saved_results(teacher_id)
    submission_id = result_record.get('submission_id')
    if submission_id:
        existing = [r for r in existing if r.get('submission_id') != submission_id]
    existing.append(result_record)
    storage_save(teacher_id, 'grading_results', existing)
```

3. **Migration consideration:** old records lack `submission_id` — harmless (they match no filter, stay in place). New records all carry it. Over time, the set converges.

4. **Regression test:** `tests/test_save_results_idempotent.py` — call `save_results` twice with same submission_id, assert exactly one record per submission_id in storage.

### Feature flag

Env var `CELERY_PORTAL_GRADING=1` enables Celery path. Default off → legacy thread path. Route reads at request time (no cached state), so a Railway env-var update + rolling restart flips the flag cleanly.

Rollback: set env to `0`, Railway rolls, next submission spawns a thread. No code change needed. In-flight Celery tasks already enqueued continue to drain — the broker doesn't know about the flag flip; the flag only gates enqueue. If the flag flip is a full rollback (not a pause), any rows stuck at `status='grading_in_progress'` after Celery drains must be manually reclaimed to the thread path (query below in monitoring).

### Enqueue-failure behavior

After the submission row is inserted (`student_portal_routes.py:781-786`), if `grade_portal_submission.delay(...)` raises (broker unreachable, Redis down):

1. **Catch the exception** at the route level
2. **Fall back synchronously** to `run_portal_grading_thread` — student gets the thread-path behavior for this one submission; submission is not lost
3. **Log a Sentry warning** with `tag=celery_enqueue_failure` so ops can see broker health regressions
4. **Return the usual 202 response** to the client — enqueue failure is transparent from the student's perspective

```python
try:
    grade_portal_submission.delay(submission_id, teacher_id, 'submissions',
                                  district_id=district_id, user_id=user_id)
except (ConnectionError, OperationalError, kombu.exceptions.OperationalError) as e:
    sentry_sdk.capture_message(
        f"Celery enqueue failed, falling back to thread: {e}",
        level='warning', tags={'celery_enqueue_failure': True},
    )
    thread = threading.Thread(
        target=run_portal_grading_thread,
        args=(submission_id, ...),
        daemon=True,
    )
    thread.start()
```

This preserves durability — a broker outage degrades to the pre-Celery experience, not to lost submissions.

#### Rollback procedure

If any monitor check shows regression:

1. **Immediate:** set `CELERY_PORTAL_GRADING=0` on Railway → rolling restart
2. New submissions spawn threads again (legacy path)
3. **Drain remaining Celery tasks:** worker keeps processing in-flight tasks until queue is empty (typically 5-15 min)
4. **Reclaim stuck rows** using query #2 above — for each row with `status='grading_in_progress'` past 30 min:
   ```sql
   UPDATE submissions
   SET status = 'queued', grading_task_id = NULL, grading_started_at = NULL
   WHERE id = '<submission_id>';
   ```
   Then trigger a manual regrade via existing teacher UI, or re-POST the submission.
5. **Investigate root cause** via Sentry + Celery worker logs before re-enabling.

## Safety rails

1. **Feature flag** — trivial rollback without code change
2. **Row-level dedup** — task_id + grading_started_at fields prevent double-grading across all retry/redelivery/crash scenarios
3. **Teacher result idempotency** — upsert-by-submission_id prevents duplicate teacher Results/Analytics on retry
4. **Explicit broker URL** — worker boot fails fast if env unset; no silent fallback to wrong infra
5. **Sentry CeleryIntegration** — task exceptions surface in existing observability pipeline with district/user tags preserved
6. **Celery `acks_late=True` + retry** — transient failures (OpenAI 503, Supabase timeout) retry automatically with exponential backoff
7. **Celery `time_limit=900`** — hard kill at 15 min prevents runaway tasks; soft limit at 14 min triggers graceful shutdown notification
8. **Explicit context in `api_keys.py`** — task can resolve API keys without Flask context; no hidden dependency on request metadata
9. **Schema gate (PR0)** — zero risk of task writing to missing columns
10. **No SIS file modifications** — Clever/ClassLink/OneRoster surface untouched; contract suite 180/180 on every PR
11. **Sequential merge required** — each PR depends on prior PR's infra

## Gotchas flagged during design (documented for implementer)

### Gotcha #1 — `run_portal_grading_thread` has TWO callers

Join-code path (`student_portal_routes.py:799`) and class-based path (`student_account_routes.py:495, 805`). Phase 4.1 migrates only the first. The wrapper function must keep working for the second path. PR3 cleanup does NOT delete the wrapper — only the join-code thread-spawn call site.

### Gotcha #2 — Gevent compatibility unverified

`assignment_grader.py` uses `ThreadPoolExecutor` + `threading.Lock` in multiple sites (`assignment_grader.py:123, 4073, 4223, 5098`). Gevent monkey-patching can interact subtly. MVP ships with prefork. Gevent is a post-MVP optimization after staging load-test.

### Gotcha #3 — Same-student double-click (pre-existing)

Two near-simultaneous `/api/student/submit/<code>` POSTs can create two distinct submission rows (`student_portal_routes.py:781-785`). This is a pre-existing race that row-level Celery dedup does NOT solve (dedup is per-submission_id; two rows = two submission_ids). Out of scope for 4.1. Documented as follow-up requiring route-level idempotency key or DB uniqueness constraint.

### Gotcha #4 — Teacher result upsert mechanism depends on storage shape

`save_results()` currently appends to teacher storage. The idempotent upsert needs to know the current storage mechanism. PR2 implementer MUST read `save_results` body to pick the minimal change. If Supabase-backed, use upsert with `submission_id` as conflict key. If file-backed (legacy `~/.graider_results.json`), delete-then-insert. Verify no regression on teacher Results/Analytics dashboards.

### Gotcha #5 — Celery prefork + Supabase client reuse

Prefork pool forks fresh worker processes. If `supabase_client.py` caches a client at module load time, forked workers inherit a client tied to the parent process's connection pool. **Mitigation:** use a per-worker client init via Celery's `worker_process_init` signal; OR confirm `supabase-py 2.27.2` handles fork-safety internally (it uses `httpx.Client` which may need explicit pool reset). PR2 implementer must verify.

### Gotcha #6 — Deployment order (broker decouples)

Worker dyno can come up before or after the web dyno flips the flag. If flag flips before worker boots, tasks queue in Redis and drain once worker comes online. No health-check gate needed (Gemini-confirmed). Document in PR3's runbook.
