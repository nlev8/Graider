# Phase 4.2 #8 — Per-student Weekly Remediation Cap

**Status:** Green-tier item from `project_phase4.2_backlog.md` #8.
**Differentiator vs Progress Learning:** PL has no per-student rate-limiting on assignments. Graider's existing `@limiter.limit("10 per minute")` on `/remediate` caps teacher abuse but does nothing to prevent the "I accidentally clicked the same red cell 4 times" scenario. This slice adds a 3-per-rolling-7-day-window per (teacher × student) cap that fails fast on /remediate AND blocks at /publish-to-class to prevent direct-API bypass.

## Goal

Teacher cannot publish more than 3 remediations to the same student in a rolling 7-day window. Hard cap (422 error). Counting basis: every publish event (including subsequently recalled ones). Cross-class: per-teacher, not per-class. Class-wide remediations count +1 against EACH targeted student. The whole publish fails if ANY targeted student is at the cap.

## Locked decisions (from brainstorming, 2026-04-30, with Codex mid-brainstorming reconciliation)

- **Q1 → A.** Hard cap, 422 error. No soft warning, no override. Recall (Phase 4.2 #5) is the existing 2-click escape hatch — recall a prior remediation if you legitimately need a 4th this week.
- **Q2 → A.** 3 per rolling 7-day window per (teacher × student). Server UTC time. NOT calendar week (timezone weirdness, Friday-Saturday edge case); NOT configurable env var (deferred — adds test surface for no current need).
- **Counting basis: INCLUDE recalled rows.** Count publish events, not currently-visible remediations. Recall is an audit/visibility action, not a quota refund — otherwise publish/recall/republish bypasses the cap (Codex MAJOR finding).
- **Class-wide counting: +1 against each targeted student.** Natural meaning of "per-student cap": every student in `target_student_ids` consumed one slot. Exempting class-wide publishes would undermine the saturation goal.
- **Cross-class scoping: per-teacher across all classes.** The harm is student saturation, not class saturation. Query filter is `teacher_id` only — no `class_id` filter.
- **Class-wide trigger surface with capped students: 422 the entire publish.** Codex MAJOR finding — partial success contradicts the hard-cap contract; "red tier except four students" is awkward state. Return capped student IDs in the 422 body so teacher knows why.
- **Cap location: BOTH /remediate AND /publish-to-class.** Codex CRITICAL finding — /remediate generates content but /publish-to-class is what writes the `published_content` row. Cap on /remediate fails fast pre-AI-call; cap on /publish-to-class prevents direct-API bypass.
- **Race condition acceptance.** Two simultaneous publishes can both see count=N and both pass. Acceptable for MVP; documented; robust fix would need DB-side transaction or RPC. The existing 10/min limiter already meaningfully rate-limits this.

## Architecture

### New module-level helper

**File:** `backend/routes/student_portal_routes.py` (alongside the existing `_validate_and_clean_lesson` helper from Phase 4.2 #1).

```python
REMEDIATION_PER_STUDENT_WEEKLY_CAP = 3  # per rolling 7-day window per (teacher × student)


def _check_remediation_cap(db, teacher_id, target_student_ids):
    """Phase 4.2 #8: returns a list of capped student IDs (those who would
    exceed the per-(teacher, student) weekly cap if a new remediation
    targeting them were published now). Empty list = clear to publish.

    Counting basis: every published_content row with target_student_ids
    containing the student, created_at >= now() - 7 days, regardless of
    is_active (recall does NOT refund the slot — it's an audit action).
    """
    from datetime import datetime, timedelta, timezone
    if not target_student_ids:
        return []
    cutoff = (datetime.now(tz=timezone.utc) - timedelta(days=7)).isoformat()

    # Fetch this teacher's recent remediations (any class, any status).
    # Mock-friendly: NO PostgREST containment operator — fetch by teacher_id
    # + cutoff and Python-filter by target_student_ids membership.
    rows = db.table('published_content').select(
        'id, target_student_ids, created_at'
    ).eq('teacher_id', teacher_id).gte('created_at', cutoff).execute()

    counts = {sid: 0 for sid in target_student_ids}
    for row in (rows.data or []):
        targets = row.get('target_student_ids')
        if not isinstance(targets, list):
            # Defensive: malformed/non-list is treated as non-counting.
            if targets is not None:
                _logger.warning(
                    "remediation.cap.malformed_target_student_ids row=%s type=%s",
                    row.get('id'), type(targets).__name__,
                )
            continue
        for sid in target_student_ids:
            if sid in targets:
                counts[sid] = counts[sid] + 1

    capped = [sid for sid, c in counts.items() if c >= REMEDIATION_PER_STUDENT_WEEKLY_CAP]
    return capped
```

### Enforcement at /remediate

Insert the cap check AFTER `target_student_ids` is fully resolved (after the single-student historical-evidence check at line ~2309 AND after the red-tier resolution at line ~2364) and BEFORE the LLM call at line ~2390. Fast-fail saves AI cost.

```python
# Phase 4.2 #8: per-student weekly cap. Block before LLM call.
capped = _check_remediation_cap(db, g.teacher_id, target_student_ids)
if capped:
    _logger.warning(
        "remediation.cap.exceeded teacher=%s class=%s standard=%s mode=%s capped=%s",
        g.teacher_id, class_id, standard_code, target_mode, capped,
    )
    return jsonify({
        "error": "Weekly remediation cap reached",
        "detail": (
            "Each student can receive at most "
            + str(REMEDIATION_PER_STUDENT_WEEKLY_CAP)
            + " remediations per rolling 7-day window."
        ),
        "capped_student_ids": capped,
        "cap": REMEDIATION_PER_STUDENT_WEEKLY_CAP,
        "window_days": 7,
    }), 422
```

The 422 response uses raw `jsonify(...)` instead of `error_response()` because `error_response()` (Phase 5d, `backend/utils/errors.py:84`) doesn't accept kwargs for extension fields — its signature is `(message, status_code, code)` only. RFC 7807 § 3.2 permits additional members; we put them in the body directly. Future work could extend `error_response()` to accept `**extra` kwargs, but that's a shared-infra change with broader blast radius — out of scope for this slice.

### Enforcement at /publish-to-class

In `backend/routes/student_account_routes.py::publish_to_class` (line ~399), apply the same cap check ONLY when the request payload has `target_student_ids` set (i.e., this is a targeted/remediation publish). Class-wide non-targeted publishes (`target_student_ids` NULL or absent) are unaffected.

```python
# Phase 4.2 #8: cap defense at the actual write path. Without this,
# direct-API callers could bypass the /remediate pre-flight check.
target_student_ids = data.get('target_student_ids')
if target_student_ids:  # non-empty list (NULL/absent = class-wide, no cap)
    # Inline-import to keep cross-route coupling minimal.
    from backend.routes.student_portal_routes import _check_remediation_cap
    capped = _check_remediation_cap(db, g.teacher_id, target_student_ids)
    if capped:
        _logger.warning(
            "publish_to_class.cap.exceeded teacher=%s capped=%s",
            g.teacher_id, capped,
        )
        return error_response(
            "Weekly remediation cap reached",
            422,
            capped_student_ids=capped,
            cap=REMEDIATION_PER_STUDENT_WEEKLY_CAP,
            window_days=7,
        )
```

### Frontend

**File:** `frontend/src/tabs/RemediationDrawer.jsx`.

The drawer's existing error handler renders `error` state (line ~205 area, `setError + setState("error")`). The 422 response includes `capped_student_ids` and `cap` in the body. Update the error display:

- If response has `capped_student_ids`, show a tailored error message: `"Weekly cap reached for N students. Recall an existing remediation to publish another, or wait until [date]."` Where N is `capped_student_ids.length` and `[date]` is computed from the oldest capped publish (server doesn't return this; frontend computes "in 7 days from oldest" by displaying a generic "wait until next week" — keep it simple).
- For non-cap errors, fall through to the existing generic `error` rendering.

The `[date]` calculation requires the server to return the oldest publish date for capped students. Out of scope for MVP — the message just says `"...or wait up to 7 days for an older remediation to age out."`

### Edge cases

| Case | Behavior |
|---|---|
| `target_student_ids = []` (empty) | `_check_remediation_cap` returns `[]` (no capped). publish_to_class already rejects empty arrays at write time. |
| `target_student_ids = None` (class-wide non-remediation publish) | Cap check NOT triggered at /publish-to-class. /remediate doesn't reach this path (target_student_ids is always populated). |
| Malformed row in DB (`target_student_ids` is not a list, e.g., string) | Skipped + logged. Doesn't cause exception. |
| Teacher recalls then publishes again | Both events count. Recall doesn't refund the slot. |
| Two simultaneous /remediate or /publish-to-class calls race past the cap | Both succeed. Acceptable for MVP. The 10/min limiter caps the practical worst case. |
| Cross-class: same student targeted from cls-1 and cls-2 | Counts as 2 against the student. Per-teacher scope. |
| Class-wide red-tier resolution returns 5 students, 2 are capped | 422 entire publish, body lists the 2 capped IDs. Teacher must drop those students from the publish (manual recall + retry, or waiting). |
| 7-day window edge: a publish that's exactly 7 days + 1 second old | Excluded (cutoff is `>= now - 7 days`, exclusive on the older side via the inequality). Acceptable. |
| Cap value = 3 means "after the 3rd publish, the 4th is blocked" | Predicate is `count >= cap` → 4th is blocked (3rd is allowed). |
| Recalled remediation row counts in cap | Yes. Recall doesn't undo the publish event. |
| Backend log volume on hot path | One `remediation.cap.exceeded` log per blocked publish. Bounded by 10/min limiter; not a log-flood risk. |
| Frontend doesn't update for new cap-related response shape | Existing error rendering catches the 422 + message. The `capped_student_ids` field is a nice-to-have for tailored UX; if frontend ignores it, generic error still shows. |

## Out of scope (deferred)

- Configurable cap via env var. Add when district admins ask.
- DB-side transaction / advisory lock to prevent simultaneous-publish races. The 10/min limiter is the practical guard for now.
- Returning the date when the oldest capped publish ages out. Frontend just says "up to 7 days" generically.
- Cap dashboard / "remediation slots remaining" UI. Out of scope; teacher discovers cap via the 422 error.
- Per-class cap separately (e.g., "no more than 5 remediations across this class per week regardless of student"). Not asked for; per-student is the documented harm.
- Soft-warn UI before hitting hard cap. Q1 → A explicitly scoped this out.
- Override / admin bypass. Q1 → A explicitly scoped this out.
- Telemetry on cap-hit rate (how often does the cap actually fire?). Existing structured logs cover this; no dedicated metric needed.
- Audit log table for cap enforcement. Existing `_logger.warning("remediation.cap.exceeded ...")` is the audit trail for now.
