---

## Goal

Add Supabase Row Level Security policies to every tenant-owned table currently missing them, and tighten policies on existing tables whose current permissions are too broad. Use the existing `auth.uid()::text = teacher_id` pattern for direct ownership and `EXISTS (SELECT 1 FROM parent ...)` for multi-hop ownership.

## Non-goal

Enforcing RLS at runtime. The Graider backend continues to use the Supabase service role key, which bypasses all RLS. Policies are belt-and-suspenders defense that become active the day Phase 4.5 introduces per-user JWT clients. **Zero production behavior change in Phase 4.2.**

---

## Hard constraints

- **SIS file lockdown** — zero modifications to `clever_routes.py`, `classlink_routes.py`, `roster_sync.py`, `oneroster_gradebook.py`, `student_account_routes.py`. SIS contract suite 180/180 green on every PR.
- **Coverage floor 32** holds (CI-enforced).
- **Boot check ≥ 250 routes** on every PR (`tests/test_app_boot.py`).
- **Backward-compatible migration** — every policy-creating statement is preceded by `DROP POLICY IF EXISTS`; every RLS-enabling statement is idempotent.
- **No `FORCE ROW LEVEL SECURITY`** — verified absent repo-wide. Service role bypass must hold.

## Design

### Enforcement model — belt-and-suspenders

- Backend uses service role key → bypasses all RLS.
- Policies written in PR1 are inert at runtime today.
- They activate the day Phase 4.5 switches the backend to per-user JWT clients.
- Until Phase 4.5 lands, policies are verified only by static + smoke tests; semantic correctness is deferred to Phase 4.5's integration test suite.

### Policy shape — EXISTS subquery for multi-hop

Direct-ownership tables use single-clause policies:

```sql
CREATE POLICY published_content_own ON published_content
  FOR ALL
  USING (auth.uid()::text = teacher_id::text)
  WITH CHECK (auth.uid()::text = teacher_id::text);
```

Multi-hop ownership uses `EXISTS` subquery:

```sql
CREATE POLICY student_submissions_own ON student_submissions
  FOR ALL
  USING (
    EXISTS (
      SELECT 1 FROM published_content
      WHERE published_content.id = student_submissions.content_id
        AND published_content.teacher_id::text = auth.uid()::text
    )
  )
  WITH CHECK (
    EXISTS (
      SELECT 1 FROM published_content
      WHERE published_content.id = student_submissions.content_id
        AND published_content.teacher_id::text = auth.uid()::text
    )
  );
```

Multi-hop correctness requires the parent table's SELECT policy to include the querying teacher. In PR1 all parent tables have `<table>_own` or equivalent that covers this.

### Naming convention — hybrid

- **Simple FOR ALL tables** keep `<table>_own` — matches existing `teacher_data_own`, `students_own`, `classes_own`. Zero churn on existing policies.
- **Split-verb tables** (`submissions`, `published_assessments`) use `<table>_<verb>_<actor>` — `submissions_select_teacher`, `submissions_insert_anonymous`, `published_assessments_update_teacher`.

### Table-by-table policy set

#### 1. `published_content` — add RLS, FOR ALL direct
```sql
ALTER TABLE published_content ENABLE ROW LEVEL SECURITY;
DROP POLICY IF EXISTS published_content_own ON published_content;
CREATE POLICY published_content_own ON published_content
  FOR ALL
  USING (auth.uid()::text = teacher_id::text)
  WITH CHECK (auth.uid()::text = teacher_id::text);
```

#### 2. `submission_confirmations` — add RLS, FOR ALL direct
Same shape as #1, `teacher_id::text` comparison.

#### 3. `student_submissions` — add RLS, FOR ALL multi-hop
EXISTS subquery through `content_id → published_content.teacher_id`.

#### 4. `class_students` — re-verify existing; already multi-hop via `class_id → classes.teacher_id`
Verify `class_students_own` policy is present; re-create idempotently with new migration for clarity.

#### 5. `student_sessions` — add RLS, FOR SELECT only
```sql
ALTER TABLE student_sessions ENABLE ROW LEVEL SECURITY;
DROP POLICY IF EXISTS student_sessions_select_teacher ON student_sessions;
CREATE POLICY student_sessions_select_teacher ON student_sessions
  FOR SELECT
  USING (
    EXISTS (
      SELECT 1 FROM classes
      WHERE classes.id = student_sessions.class_id
        AND classes.teacher_id::text = auth.uid()::text
    )
  );
```
No INSERT/UPDATE/DELETE policies — writes by non-service-role are implicitly denied. Backend (service role) still performs all session writes.

#### 6. `submissions` — tighten to split policies

Current state: "Anyone can insert", "Anyone can read", service_role full access. All three broad policies get dropped; replaced with scoped equivalents:

```sql
DROP POLICY IF EXISTS "Anyone can insert submissions" ON submissions;
DROP POLICY IF EXISTS "Anyone can read submissions" ON submissions;

-- Teacher can read submissions to their own assessments
DROP POLICY IF EXISTS submissions_select_teacher ON submissions;
CREATE POLICY submissions_select_teacher ON submissions
  FOR SELECT
  USING (
    EXISTS (
      SELECT 1 FROM published_assessments
      WHERE published_assessments.id = submissions.assessment_id
        AND published_assessments.teacher_id = auth.uid()::text
    )
  );

-- Teacher can update/delete submissions to their own assessments
DROP POLICY IF EXISTS submissions_update_teacher ON submissions;
CREATE POLICY submissions_update_teacher ON submissions
  FOR UPDATE
  USING (
    EXISTS (
      SELECT 1 FROM published_assessments
      WHERE published_assessments.id = submissions.assessment_id
        AND published_assessments.teacher_id = auth.uid()::text
    )
  );

DROP POLICY IF EXISTS submissions_delete_teacher ON submissions;
CREATE POLICY submissions_delete_teacher ON submissions
  FOR DELETE
  USING (
    EXISTS (
      SELECT 1 FROM published_assessments
      WHERE published_assessments.id = submissions.assessment_id
        AND published_assessments.teacher_id = auth.uid()::text
    )
  );

-- No INSERT policy for non-service-role — backend-only writes.
-- Anonymous join-code submission path goes through backend (service role bypass).
```

Service role policy already exists and is preserved.

#### 7. `published_assessments` — tighten to teacher-only + drop anonymous read

Current state: "Anyone can read active assessments" + service_role. The anonymous-read policy is dropped (zero production usage; backend routes the join-code flow).

```sql
DROP POLICY IF EXISTS "Anyone can read active assessments" ON published_assessments;

DROP POLICY IF EXISTS published_assessments_select_teacher ON published_assessments;
CREATE POLICY published_assessments_select_teacher ON published_assessments
  FOR SELECT
  USING (teacher_id = auth.uid()::text);

DROP POLICY IF EXISTS published_assessments_insert_teacher ON published_assessments;
CREATE POLICY published_assessments_insert_teacher ON published_assessments
  FOR INSERT
  WITH CHECK (teacher_id = auth.uid()::text);

DROP POLICY IF EXISTS published_assessments_update_teacher ON published_assessments;
CREATE POLICY published_assessments_update_teacher ON published_assessments
  FOR UPDATE
  USING (teacher_id = auth.uid()::text)
  WITH CHECK (teacher_id = auth.uid()::text);

DROP POLICY IF EXISTS published_assessments_delete_teacher ON published_assessments;
CREATE POLICY published_assessments_delete_teacher ON published_assessments
  FOR DELETE
  USING (teacher_id = auth.uid()::text);
```

Service role policy preserved.

#### 8-11. `teacher_data`, `student_history`, `classes`, `students` — re-verify

Existing `<table>_own` policies confirmed via direct schema inspection. Migration re-creates them idempotently (DROP + CREATE) for documentation parity; no semantic change.

## Migration plan

**Migration file:** `backend/database/migration_2026_04_17_phase4.2_rls.sql`
**Rollback file:** `backend/database/rollback_2026_04_17_phase4.2_rls.sql`

Migration ordered sections:
1. Preamble comment — purpose, enforcement model, Phase 4.5 handoff pointer, rollback instructions
2. Re-verify existing `<table>_own` policies (idempotent DROP + CREATE)
3. Parent-table new or tightened policies (published_content, published_assessments)
4. Child-table multi-hop policies (submissions, student_submissions, student_sessions, class_students)
5. Direct-ownership additions (submission_confirmations)

Rollback contents:
- `DROP POLICY IF EXISTS` for every policy PR1 adds
- `ALTER TABLE ... DISABLE ROW LEVEL SECURITY` for the 4 new-RLS tables (`student_sessions`, `published_content`, `student_submissions`, `submission_confirmations`)
- Does NOT disable RLS on existing-RLS tables (`submissions`, `published_assessments`); those revert to pre-PR1 policy state via the DROPs

Application path: Supabase SQL editor or `supabase db push` (same as Phase 4.1 PR0 convention — Graider has no automated migration runner; Phase 4.3 tracks that gap).

## Test strategy

### Static test — `tests/test_rls_policy_coverage.py`

Uses `pglast` (libpg_query Python wrapper) to parse all schema SQL + new migration. Asserts for every PR1 table:

1. `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` statement exists
2. At least one `CREATE POLICY ... ON <table>` exists
3. Every policy's USING / WITH CHECK clause references `auth.uid()` OR `EXISTS` (shape check)

Catches: missing RLS enable, missing policy, policy with no auth check.
Misses: policy semantic correctness (requires real JWT).

Adds `pglast==6.*` to `requirements-dev.txt`. Module docstring documents the maintained table list and notes the requirement to update the list when adding new tenant tables.

### Smoke test — `tests/test_rls_migration_applies.py`

Uses `testcontainers[postgres]` to spin up Postgres 15 inside the pytest process. No GitHub Actions workflow changes required (Docker is available on GitHub-hosted runners).

Test flow:
1. Start Postgres 15 container
2. Create minimal `auth` schema stub: `CREATE SCHEMA auth; CREATE OR REPLACE FUNCTION auth.uid() RETURNS text AS $$ SELECT NULL::text; $$ LANGUAGE sql;`
3. Apply base schema files: `backend/database/supabase_schema.sql`, `supabase_student_portal_schema.sql`, `supabase_submission_confirmations.sql`, `backend/database/supabase_teacher_schema.sql`, `supabase_roster_rls.sql`
4. Apply PR1 migration
5. Assert expected policy names exist in `pg_policies` for each PR1 table
6. Assert `pg_tables.rowsecurity = true` for the 4 new-RLS tables

Catches: SQL syntax errors, forgotten table references, EXISTS subquery mistakes, column reference bugs, policy count mismatches.
Misses: policy semantic correctness (same limit as static test).

Adds `testcontainers[postgres]==4.*` to `requirements-dev.txt`.

### Deferred integration test — Phase 4.5 scope

When per-user JWT lands, add `tests/test_rls_semantic_enforcement.py` that:
1. Connects to a dev Supabase project with a test user's JWT (not service role)
2. For each PR1 table, verifies SELECT returns only own rows
3. For each PR1 table, verifies INSERT / UPDATE / DELETE of other teachers' rows fails with an RLS violation
4. For multi-hop tables, verifies parent-deletion mid-query returns empty rather than errors

Gated by `SUPABASE_RLS_TEST_JWT` env var — skipped when missing (local dev works without credentials). Requires one-time dev Supabase test user provisioning.

## Phase 4.5 handoff contract

When Phase 4.5 introduces per-user JWT clients for the backend, the integration tests must cover:

**Per-table direct ownership:**
- Teacher A's JWT: can read / insert / update / delete only rows where `teacher_id = auth.uid()::text`
- Teacher A's JWT: cannot read teacher B's rows, even if the row ID is known
- Anonymous JWT (no sub claim): no access to any teacher-owned tables

**Per-table multi-hop ownership (`submissions`, `student_submissions`, `student_sessions`, `class_students`):**
- Teacher A's JWT: can CRUD child rows only when parent row's `teacher_id = auth.uid()::text`
- Teacher A's JWT: attempting to read child rows whose parent was deleted must return empty result (not SQL error)
- Teacher A's JWT: attempting to INSERT a child row pointing to teacher B's parent must fail

**Specific tables to watch:**
- `student_sessions` — read-only under teacher's JWT; all writes must go through service role
- `submissions` — INSERT policy deliberately absent for teacher JWTs; anonymous join-code submission path stays on the backend (service role)
- `published_assessments` — anonymous read policy deliberately absent; the join-code flow stays on the backend

If any of these invariants fail when Phase 4.5 cutover runs, the Phase 4.2 policies need revision. Document the specific failure in a Phase 4.5 follow-up.

## Risks

1. **Schema drift on live Supabase.** Graider has no migration runner; the live DB may have policies or tables the repo SQL files don't reflect. Smoke test passes (clean fixture) but live migration could fail or leave stale policies behind. **Mitigation:** Phase 4.3 (migration tooling) is the structural fix; Phase 4.2 applies ad-hoc by verifying `pg_policies` in the live DB after applying the migration.
2. **Mis-tightened `submissions` or `published_assessments` policy.** A wrong policy could block a teacher-facing read in the Phase 4.5 cutover — invisible in PR1 because service role bypasses. **Mitigation:** the Phase 4.5 integration test suite catches this before the JWT cutover lands. Documented in the handoff contract above.
3. **`pglast` library pin stability.** `pglast 6.*` depends on bundled libpg_query C extension; install can fail on unusual platforms. **Mitigation:** fallback — if `pglast` import fails in CI, the static test can be skipped with a `pytest.importorskip`. Flagged as a sharp edge, not a blocker.
4. **Multi-hop policy silent failure modes.** An EXISTS subquery that evaluates against a parent table whose own RLS policy is broken returns empty. The Phase 4.2 audit confirmed all parent tables have teacher-own SELECT policies, but any future change to a parent's policies could break child access invisibly.
