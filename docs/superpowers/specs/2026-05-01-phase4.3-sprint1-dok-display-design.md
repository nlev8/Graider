## Architecture

Three concerns, three wire-contract additions, one new shared helper, one new metadata line in the drawer:

1. **Per-question DOK pass-through (write side).** AI generates `dok` per question → JSONB at `published_content.content.questions[i].dok`. The grading writer in `portal_grading.py` builds `student_submissions.results.questions` with a fixed field set and currently DROPS `dok`. This sprint extends the writer to preserve `dok` from source content to results.

2. **Per-question DOK pass-through (read side).** The SubmissionDetail per-question response normalizer at `student_portal_routes.py:2100-2109` also normalizes a fixed field set and drops unknown keys. Extend it to surface `dok` to the drawer.

3. **Uniform DOK derivation (gradebook + drawer header).** Both surfaces today render `RemediationBadges` (Phase 4.2 #7) using `is_active` + `target_student_ids` flags. Extend the gradebook list route + drawer header route to additionally derive a uniform DOK from the source `content.questions[*].dok` field for remediation rows only. Pattern: when `target_student_ids IS NOT NULL` AND all questions share the same `dok` value AND that value is `1..4`, emit `assessment_dok: N`. Otherwise emit `null` and the badge component renders no DOK pill.

The frontend extends one component (`RemediationBadges.jsx`) to read a new `item.assessment_dok` field and render a third pill conditionally. The SubmissionDetail drawer also adds a small new metadata line under each question text (the card currently shows question text + points but has no first-row metadata).

## Wire contract

### Gradebook list response

Each entry in the `assessments` array gains:

```diff
{
  "id": "...",
  "title": "...",
  "content_type": "assessment",
  "publish_date": "...",
  "due_date": "...",
  "is_active": true,
  "target_student_ids": ["...", "..."],
+ "assessment_dok": 3        // int 1..4 when remediation + uniform; null otherwise

```

### SubmissionDetail response

Top-level header gains `assessment_dok`. Per-question entries gain `dok`:

```diff
{
  "submission_id": "...",
  "student_id": "...",
  "content_id": "...",
  "content_title": "...",
  "is_active": true,
  "target_student_ids": ["..."],
+ "assessment_dok": 3,
  "questions": [
    {
      "question_text": "...",
      "question_type": "multiple_choice",
      "student_answer": "...",
      "correct_answer": "...",
      "is_correct": true,
      "ai_feedback": "...",
      "points_earned": 5,
      "points_possible": 5,
+     "dok": 3              // int 1..4 when source content has it; null otherwise
    }

}
```

`RemediationBadges` reads `item.assessment_dok` directly (no prop threading at call sites).

## Components

### New backend helper: `_validate_dok(value)`

Location: `backend/routes/student_portal_routes.py` (alongside existing constants like `DOK_OPTIONS`).

Defends against:
- Stored legacy data with no `dok` key
- AI output drift (string `"3"` instead of int `3`)
- Boolean coercion (`bool` is `int` subclass — `True` would slip past naive `isinstance(int)`)

Contract:

```python
def _validate_dok(value):
    """Normalize a dok value to an int in DOK_OPTIONS (1..4) or None.

    Accepts: int 1-4, numeric string "1"-"4".
    Rejects: bool (any), None, non-numeric strings, ints outside 1-4,
             floats, anything else.

    Codex round 1 MINOR — frontend renders only int 1-4, but legacy
    storage / model output drift can produce string DOKs. Normalize
    on read to keep the frontend predicate simple.
    """
```

**`backend/routes/student_portal_routes.py`**:

- **SubmissionDetail route** (around the questions builder + final `jsonify` at line 2114):

  - Derive `assessment_dok` from `content.questions[*].dok` using the same uniform-DOK rule.
  - Add `"assessment_dok": assessment_dok` to the response dict at line ~2114, alongside `is_active` / `target_student_ids`.

- **Per-question payload** (lines 2100-2109): add `"dok": _validate_dok(q.get('dok'))` to the `questions.append(...)` payload. This pulls from `student_submissions.results.questions` (which now has `dok` thanks to the portal_grading fix) and normalizes it.

## Tests

### Backend

`tests/test_remediation.py` (existing file, extend):

1. `TestPortalGradingDokPassthrough` — single test verifying that when source content has `dok=3` per question, `student_submissions.results.questions[i].dok == 3` after grading. Smoke test on the write side; uses a mocked `published_content` row.

2. `TestSubmissionDetailDokFields` — test the response normalizer:
   - Source results.questions has `dok=3` → response per-question has `dok: 3`
   - Source results.questions missing `dok` → response per-question has `dok: None`
   - Source results.questions has `dok="3"` (string) → response normalizes to `3` via `_validate_dok`

3. `TestSubmissionDetailHeaderAssessmentDok` — test header derivation:
   - Uniform DOK 3 across all questions → `assessment_dok: 3`
   - Mixed DOK (some 2, some 3) → `assessment_dok: null`
   - All questions missing `dok` → `assessment_dok: null`

4. `TestGradebookAssessmentDok` — test gradebook uniform-DOK derivation:
   - Class with 1 remediation (uniform DOK 2) + 1 regular assessment → remediation row has `assessment_dok: 2`, regular row has `assessment_dok: null`
   - Class with 1 remediation (mixed DOK) → remediation row has `assessment_dok: null`
   - Class with 0 remediations → all rows have `assessment_dok: null`

### `_validate_dok` helper

5. `TestValidateDok` — unit tests:
   - `_validate_dok(2)` → `2`
   - `_validate_dok("3")` → `3`
   - `_validate_dok(True)` → `None` (bool rejected before int check)
   - `_validate_dok(0)` → `None` (out of range)
   - `_validate_dok(5)` → `None`
   - `_validate_dok("foo")` → `None`
   - `_validate_dok(None)` → `None`

### Frontend

Manual smoke checklist (matches Phase 4.2 precedent — no Vitest/RTL):

- Generate a remediation with DOK 3 → open Gradebook → see "DOK 3" pill on the column header alongside "Remediation"
- Open SubmissionDetail drawer → see "DOK 3" pill in the header alongside "Remediation"
- Click into a graded submission → each question card shows the new metadata line "Q1 · multiple_choice · 5 pts · DOK 3"
- Submit a non-remediation Planner assessment with mixed DOK → no DOK pill in Gradebook column header, no DOK on per-question cards (legacy/non-remediation graceful fallback)
- Recall a remediation → both "Remediation" + "Recalled" + "DOK 3" pills render together
- A legacy remediation generated before Phase 4.2 #12 (no `dok` field) → no DOK pill, no per-question DOK token (graceful fallback)

## Backward compatibility

- Legacy submissions (pre-#12, no `dok` in source content): `_validate_dok(None) → None` → frontend renders nothing. No broken UI.
- Mixed-DOK Planner content: `assessment_dok = null` → no badge. Range/distribution badge is YAGNI for this sprint.
- AI output drift (`dok` as string): normalized via `_validate_dok` on read.
