# Phase 4.2 — Remediation Effectiveness Dashboard (Design)

**Status:** Yellow-tier item from `project_phase4.2_backlog.md` #6.
**Differentiator vs Progress Learning:** PL doesn't tie effectiveness measurement to specific remediation publish events — they show generic mastery progression filterable by date. Graider can attribute mastery delta to specific remediation events because we know which `published_content` rows were remediations (`target_student_ids IS NOT NULL`).

## Goal

Teacher answers "did my recent remediations work?" at a glance + acts on gaps in one click. Per-(student × remediation) row showing before/after mastery delta + completion + attempt count, with a "Remediate again" CTA per row.

## Locked decisions (from brainstorming, 2026-04-27)

- **Q1 → A.** New 4th sub-tab in `AnalyticsTab.jsx` next to Progress Rank / Gradebook / Compare. Class-scoped (uses existing class selector). Conditional unmount pattern (matches Phase 3a/3b).
- **Q2 → B.** Three metrics per row: mastery delta on the targeted standard (headline) + completion (binary, did the student take the remediation) + attempt count.
- **Q3 → A.** Identify a remediation via `target_student_ids IS NOT NULL` on `published_content`. No schema change. Documented in code as a convention (today only `/api/teacher/class/<id>/remediate` populates this field; if another feature ever publishes to a class subset, add an `is_remediation` flag at that time).
- **Q4 → B.** "Before" baseline = latest pre-remediation submission's `results.standards_mastery[standard_code].percentage` for this student. Computed at query time; no baseline-snapshot storage. Always-fresh from rollups (matches Phase 2/2b/3a/3b pattern).
- **Q5 → A.** "After" mastery = latest submission for this student that includes this standard in its `standards_mastery` rollup, regardless of which content row produced it. Reflects current mastery whether they revisited via the remediation or got it through another assessment.
- **Q6 → A.** No time filter. Show all remediations published to the selected class. Phase 4.3 can add a date-range picker if classes accumulate >20 rows.
- **Q7 → B.** Per-row "Remediate again" CTA opens the existing `RemediationDrawer` (Phase 4) in single-student mode pre-filled with this row's `(student_id, standard_code)`. Closes the loop without forcing a context switch back to ProgressRankGrid.

## Architecture

### Backend — one new route

**Route:** `GET /api/teacher/class/<class_id>/remediation-effectiveness` in `backend/routes/student_portal_routes.py`. Decorators: `@require_teacher`, `@handle_route_errors`. Class-scoped (path-style URL matches Phase 2/2b/3a/3b).

**Validation:**
1. Class ownership (`classes.teacher_id == g.teacher_id`) → 403 via `error_response`.
2. No body / no other inputs (read-only GET).

**Query plan:**
1. Fetch all `published_content` rows for this class with `target_student_ids IS NOT NULL`. Select: `id, title, content_type, created_at, target_student_ids, settings, content`.
2. Extract the targeted standard from each remediation row. Stored where? See "Targeted standard storage" below.
3. Build the union set `all_target_student_ids` = union of all target_student_ids arrays across remediations.
4. Fetch `students` rows for `all_target_student_ids` to get student names.
5. Fetch `student_submissions` for these students with `student_id IN all_target_student_ids`, `status != 'draft'`, ordered by `submitted_at` desc. (Same pattern as Phase 3b's red-tier resolver — fetch broad, post-filter on `results.standards_mastery` in Python.)
6. For each `(remediation, student_id)` pair (expand `target_student_ids` array):
   - **Before:** scan submissions where `submitted_at < remediation.created_at` and `results.standards_mastery[standard_code]` exists. Take the latest. Read `mastery.percentage`. If none, `before = NULL`.
   - **After:** scan submissions where `results.standards_mastery[standard_code]` exists (no time filter; always-current). Take the latest. Read `mastery.percentage`. If none, `after = NULL`.
   - **Delta:** `after - before` if both non-null; else `NULL`.
   - **Completed:** any submission where `student_id == this student AND content_id == this remediation.id AND status != 'draft'`.
   - **Attempt count:** count of those submissions.
7. Return JSON.

**Response shape:**

```json
{
  "class_id": "uuid",
  "class_name": "Period 3",
  "remediations": [
    {
      "remediation_id": "uuid",
      "title": "Practice — MA.6.AR.1.2",
      "standard_code": "MA.6.AR.1.2",
      "created_at": "2026-04-26T18:00:00Z",
      "target_count": 3,
      "rows": [
        {
          "student_id": "uuid",
          "student_name": "Alex C.",
          "before": 40.0,        // null if no baseline
          "after": 78.5,         // null if no post-remediation evidence
          "delta": 38.5,         // null if before or after is null
          "completed": true,
          "attempt_count": 1
        },
        ...
      ]
    },
    ...
  ]
}
```

### Targeted standard storage

The route needs to know which standard each remediation targeted. Options:

1. **Read from `settings.target_standard`** — add a write at remediation publish time.
2. **Infer from `content.questions[0].standard`** — every remediation question has `standard` field set to the targeted code (Phase 4 invariant); read the first question. Brittle if AI ever returns a question with a different standard.
3. **Infer from `content.title`** — the title is `"Practice — {standard_code}"` per Phase 4 prompt. Title-parsing is fragile.

**Decision:** option 1. Add a write to `published_content.settings.target_standard` from the publish-call site. The remediation publish flow originates in `RemediationDrawer.jsx::publish()` which calls `api.publishToClass(...)` — it knows the standard_code from the route response. Pass it via the `settings` field of the publish body. No schema change (settings is already JSONB).

**Implementation note:** the existing `RemediationDrawer.jsx::publish()` builds the publish payload but doesn't currently include `settings.target_standard`. This spec adds it. No new column.

For backwards compatibility with any pre-existing remediations (none in production yet, since Phase 4 just shipped), the route falls back to option 2 (inferring from `content.questions[0].standard`) when `settings.target_standard` is missing.

### Frontend — one new component + sub-tab integration

**New file:** `frontend/src/tabs/RemediationEffectiveness.jsx` (~250 LOC). Receives `classId` prop. Owns its own fetch lifecycle.

**Layout:**
- Header: "Remediation Effectiveness" + class name subtitle. Optional: small description "Did your recent remediations move student mastery?"
- Body: scrollable list of remediation event cards. Each card has:
  - Title + standard code (monospace badge) + date published.
  - Table inside the card with columns: Student | Before % | After % | Δ | Completed | Attempts | Action.
  - Sticky student name column (matches Phase 3a Gradebook pattern).
  - Δ cell: color-coded (≥+10 green, 0..+9 yellow, <0 red, gray for NULL).
  - "Remediate again" button per row → opens `RemediationDrawer` with `target_mode='single_student'`, `target_student_id=row.student_id`, `standard_code=remediation.standard_code`, `targetStudentName=row.student_name`.
- Empty state: "No remediations published yet. Use the Progress Rank grid to publish one." with subtle link to the Progress Rank sub-tab.

**Sub-tab integration in `AnalyticsTab.jsx`:**
- Add 4th button "Effectiveness" alongside Progress Rank / Gradebook / Compare.
- Update `classView` enum: `'progressRank' | 'gradebook' | 'compare' | 'effectiveness'`.
- Replace the existing 3-way ternary with a 4-way `if/else if` chain (cleaner than nested ternary at 4 conditions) OR continue nested ternary (matches existing style). Implementer picks based on consistency.
- Sub-tab strip styling: add `overflow-x: auto` and `flex-wrap: nowrap` so 4 buttons stay on one line on narrow viewports (~600px). Existing strip uses `display: flex` with no overflow handling — Codex round 1 of brainstorming flagged this as minor crowding risk.

**Drawer reuse:**
- The `RemediationDrawer` component needs no changes. It already accepts the props this dashboard provides.
- On drawer's `onPublished` callback, the dashboard re-fetches its data (refresh pattern matches ProgressRankGrid Phase 4.2 fix).

### Edge cases (per Codex mid-check)

| Case | UX |
|---|---|
| No pre-remediation submission for the standard | "Before" cell shows `"No baseline"` badge; Δ cell shows `"—"` |
| Student didn't take the remediation (`completed: false`) | After column shows the latest submission with this standard regardless (per Q5 → A); if none, `"—"` |
| Multiple remediations for same (student, standard) | Each `published_content` row is its own remediation card; same student appears in multiple cards in chronological order |
| Class has zero remediations | Empty state: "No remediations published yet. Use the Progress Rank grid to publish one." |
| `target_student_ids IS NOT NULL` is a CONVENTION not a constraint | Comment in route handler + dashboard query: `"# Today, target_student_ids != NULL implies a remediation. If another feature ever publishes to a class subset, add an is_remediation flag and update this filter."` |
| `settings.target_standard` missing on old rows | Fall back to `content.questions[0].standard` if available; if neither exists, exclude the remediation from the dashboard with a `_logger.warning("remediation.unknown_standard remediation_id=%s", id)` log line |

### Testing

**Backend** — ~12 tests in new `tests/test_remediation_effectiveness.py`:

| Class | Count | Coverage |
|---|---|---|
| `TestEffectivenessAuth` | 2 | 401 unauth, 403 wrong teacher |
| `TestEffectivenessHappyPath` | 4 | Empty class (no remediations), single remediation × single student improved (Δ>0), single remediation × single student didn't improve (Δ<0), multiple remediations same student same standard distinct rows |
| `TestEffectivenessEdgeCases` | 4 | NULL baseline (no pre-remediation submission), student didn't complete (completed=False), multiple students per remediation, target_standard fallback to content.questions[0].standard |
| `TestEffectivenessIsolation` | 2 | Other class's remediations not leaked, non-remediation published_content (target_student_ids IS NULL) excluded |

**Filter-aware mock:** reuse the `_make_chain` pattern from `tests/test_assessment_comparison.py` (records `.eq()`, `.in_()`, `.neq()`, `.range()` filters and applies them at `.execute()` time).

**Frontend:** build verification only (consistent with Phase 2/2b/3a/3b/4 precedent).

**Sibling regression:** `tests/test_gradebook.py`, `tests/test_submission_detail.py`, `tests/test_student_report_card.py`, `tests/test_assessment_comparison.py`, `tests/test_remediation.py`, `tests/test_student_account_coverage.py`, `tests/test_student_resources.py` — all must continue passing.

### Compliance

Reads only `classes`, `class_students`, `students`, `published_content`, `student_submissions`. No schema migration. No writes (read-only GET). Zero touch to Clever / ClassLink / OneRoster / LTI surfaces. No PII in dashboard payload beyond what existing teacher analytics already expose.

### Sequencing

This spec maps to **one PR** with the following commits:

1. **`RemediationDrawer.jsx::publish()` writes `settings.target_standard`** in the publish payload (small frontend change so going-forward remediations have the field).
2. **Backend route** + 12 tests in `tests/test_remediation_effectiveness.py`.
3. **Frontend API client** function + `RemediationEffectiveness.jsx` component.
4. **AnalyticsTab.jsx** wire-up (4th sub-tab + 4-way render + overflow CSS).

Estimated scope: ~150 LOC backend route + ~250 LOC frontend + ~250 LOC tests. ~3-4 hours to implement + review + ship.

### Out of scope (Phase 4.3 backlog candidates)

- Date-range filter (Progress Learning's pattern). Add when classes accumulate >20 remediations.
- "Show only ineffective" triage filter (delta <0 OR `completed: false`).
- Cross-class effectiveness rollup (school-wide view).
- Per-question pass-rate on the practice itself (≥85% on the remediation submission).
- Time-to-completion metric (gap between publish and student's first attempt).
- Configurable Δ thresholds (currently hardcoded ≥+10 green / 0..+9 yellow / <0 red).
- Export to CSV.
- Bulk "remediate all ineffective" action.
- "Effectiveness over time" line chart per remediation event (mastery trajectory).

### Risk

**Low.** Read-only endpoint reusing existing rollup helpers. One small additive write (`settings.target_standard`) on the publish path that's backwards-compatible (route falls back to inferring from `content.questions[0].standard`). Frontend pattern matches Phase 3a/3b/4 verbatim. No schema migration.
