# Phase 4.2 #2 — Per-student Generation in Class-wide Remediations

**Status:** Yellow-tier item from `project_phase4.2_backlog.md` #2 — the deferred Q5 → C from Phase 4.
**Differentiator vs Progress Learning:** PL has no per-student personalization in class-wide remediations. Graider's Phase 4 single-student mode already honors accommodations; this slice extends that to class-wide red-tier mode when at least one targeted student has accommodations on file.

## Goal

When `target_mode='red_tier_in_class'` resolves to N students AND at least one has accommodations, `/remediate` generates N personalized assessments (one per student) using each student's accommodation data. The drawer presents N variants in tabs (one per student); publishing creates N `published_content` rows in a single atomic batch. If all N students lack accommodations, the existing shared-generation path is used unchanged. If any student is at the per-student weekly cap (#8), or any AI generation fails, the entire publish fails.

## Locked decisions (from brainstorming + Codex mid-brainstorming reconciliation, 2026-04-30)

- **Q1 → C.** Auto-decide trigger. The route loads each targeted student's accommodation data via `build_accommodation_prompt(student_id, teacher_id)`. If ANY student returns a non-empty string, switch to personalized mode. Otherwise stay on the existing shared path. **Codex MINOR caveat (accepted):** `build_accommodation_prompt` returns delimiter-only output for some empty-preset edge cases — accept that as "has accommodation" rather than building stricter semantic detection.
- **Q2 → A.** Hardcoded cap at 10 students max in personalized mode. If red-tier resolves to >10 students with accommodations, return 422 ("Class has too many students for personalized mode; use single-student mode for individual students or remove accommodations from some"). Constant `REMEDIATION_PERSONALIZED_MAX = 10` in `student_portal_routes.py`.
- **Q3 (failure handling):** All-or-nothing. **Codex CRITICAL reconciliation:** N independent drawer publish calls cannot deliver all-or-nothing — switching to a new batch endpoint (below) makes atomic write possible.
- **Q4 (uniform N):** Even targeted students without accommodations get their own personalized row in personalized mode. Their prompt simply omits the accommodation_segment. NOT a hybrid where students without accommodations share one row.
- **Q5 (parallelism):** `ThreadPoolExecutor(max_workers=min(N, 5))`. **Codex MAJOR reconciliation:** Cap to 5 concurrent OpenAI calls (not N) to reduce nondeterministic rate-limit failures. With all-or-nothing semantics, any single failure aborts the whole publish; lower concurrency = lower failure probability.
- **Q5b (per-thread adapter):** Each worker creates its own `OpenAIAdapter` instance. **Codex MAJOR reconciliation:** Defensive — adapter holds a shared `OpenAI` client; instantiating per call avoids any shared-state surprise. Object-allocation cost is negligible.
- **Q6 (drawer UX):** Tabs (one per student name). Click student to switch the preview pane. Matches the SubmissionDetail attempt-selector precedent.
- **Q7 (storage):** N separate `published_content` rows, each `target_student_ids=[that one student]`. Aligns with effectiveness dashboard cards, recall semantics, weekly cap counting — all per-row.
- **Q8 (publish flow) — NEW:** Drawer issues ONE call to a new `POST /api/publish-to-class-batch` endpoint that accepts an array of `{content, target_student_ids, ...}` and writes them in a SINGLE atomic batch. **Codex CRITICAL reconciliation:** Replaces the original "N drawer calls" plan to make all-or-nothing semantics actually true. Net LOC neutral (one batch endpoint vs the partial-success state machine the alternative would require).
- **Variant ordering:** Variants sorted by student name (alphabetical) in /remediate response and rendered in tab order. **Codex MAJOR reconciliation:** ThreadPoolExecutor returns results in nondeterministic completion order; UX requires deterministic ordering.
- **Cap preflight at batch publish:** Batch endpoint runs `_check_remediation_cap` BEFORE writing — same posture as Phase 4.2 #8 enforcing on both `/remediate` and `/publish-to-class`. **Codex MAJOR reconciliation:** time passes between /remediate and publish; cap state may have changed.
- **Race condition acceptance:** Unchanged from Phase 4.2 #8 — two simultaneous publishes can pass cap check; transactional cap enforcement deferred.

## Architecture

### Worker-thread context (Codex round 2 MAJOR)

Flask's `g` is request-scoped and NOT safe to read from worker threads. Before submitting work to the ThreadPoolExecutor, capture all needed values into local variables and pass as plain args. The captured set:

- `teacher_id` (from `g.teacher_id`)
- `api_key` (from `_gak('openai', g.teacher_id)`)
- `_ctx_uid`, `_ctx_client` (from `_get_openai_context()`)
- `subject`, `grade`, `standard_code`, `base_prompt` (already locals)
- `students_by_id` — a `{student_id: {first_name, last_name}}` dict — fetched ONCE via `db.table('students').select('id, first_name, last_name').in_('id', target_student_ids).execute()` before submitting to the pool. Resolves Codex round 2 MINOR on student-name source.
- `per_student_segments` (already a local dict)

The worker function takes ONLY these plain values as parameters; it never references `g`, `request`, `db`, or any Flask globals.

### Backend — `/remediate` route (modified)

**File:** `backend/routes/student_portal_routes.py` (existing route at line ~2208).

After target_student_ids is resolved (existing single-student or red-tier paths) AND after the per-student weekly cap check (Phase 4.2 #8), branch on accommodation detection:

```python
# Phase 4.2 #2: auto-decide whether to personalize. Personalize iff at
# least one targeted student has a non-empty accommodation_segment.
# In single-student mode, target_student_ids has length 1 — the existing
# accommodation_segment logic already covers it; we keep it on the shared
# path. Personalization is a class-wide-mode-only branch.
should_personalize = False
per_student_segments = {}  # sid -> str (may be empty)
if target_mode == 'red_tier_in_class' and len(target_student_ids) > 1:
    if len(target_student_ids) > REMEDIATION_PERSONALIZED_MAX:
        # Will only block if any student has accommodations; otherwise
        # the shared path is fine for >10 students.
        # Check accommodations first to know which path applies.
        pass  # logic below
    for sid in target_student_ids:
        try:
            seg = build_accommodation_prompt(sid, g.teacher_id) or ""
        except Exception:
            _logger.warning(
                "remediation.personalize.accommodation_load_failed teacher=%s student=%s",
                g.teacher_id, sid,
            )
            seg = ""
        per_student_segments[sid] = seg
        if seg:
            should_personalize = True

    if should_personalize and len(target_student_ids) > REMEDIATION_PERSONALIZED_MAX:
        return jsonify({
            "error": "Class has too many students for personalized mode",
            "detail": (
                "Personalized remediation supports at most "
                + str(REMEDIATION_PERSONALIZED_MAX)
                + " students at a time. Use single-student mode for "
                + "individual students, or remove accommodations from "
                + "some students to fall back to shared mode."
            ),
            "target_count": len(target_student_ids),
            "max": REMEDIATION_PERSONALIZED_MAX,
        }), 422
```

If `should_personalize` is False, fall through to existing shared-generation path (unchanged behavior).

If True, generate N variants in parallel:

```python
from concurrent.futures import ThreadPoolExecutor
import threading

def _gen_variant_for_student(*, sid, segment, students_by_id, api_key,
                             base_prompt, subject, grade, standard_code,
                             ctx_uid, ctx_client, teacher_id, class_id):
    """Runs in a worker thread. Builds prompt, calls AI, post-processes,
    returns a {student_id, student_name, questions, lesson, usage} dict
    or raises.

    Per-thread OpenAIAdapter instance to avoid shared-state concerns.
    All Flask globals (g, request) and DB handles are forbidden in this
    function — only the explicit kwargs above are accessible (Codex
    round 2 MAJOR on worker-thread context).

    Note: cost telemetry (usage) is RETURNED rather than recorded inside
    the worker. The parent thread aggregates and records once after all
    variants complete (Codex round 2 MINOR — _record_planner_cost is
    read-modify-write JSON without locking).
    """
    from backend.services.llm_adapter import OpenAIAdapter, LLMRequest, Message, ResponseFormat, TextPart
    adapter = OpenAIAdapter(api_key=api_key)
    prompt = base_prompt + ("\n\n" + segment if segment else "")
    completion = adapter.chat(LLMRequest(...))
    raw = _json.loads(completion.content_parts[0].text)
    # ... existing fallback wrapper + _post_process_assignment ...
    # _post_process_assignment is invoked with ctx_uid + ctx_client (NOT g).
    flat_questions = [q for sec in (assignment or {}).get('sections', []) for q in sec.get('questions', [])]
    if len(flat_questions) < 3:
        raise ValueError(f"Generation produced too few valid questions for student {sid}")
    s_row = students_by_id.get(sid) or {}
    student_name = ((s_row.get('first_name') or '') + ' ' + (s_row.get('last_name') or '')).strip() or sid
    return {
        'student_id': sid,
        'student_name': student_name,
        'questions': flat_questions,
        'lesson': _validate_and_clean_lesson(assignment.get('lesson'), ...),
        'usage': _extract_usage(completion, "gpt-4o"),
    }

variants = []
errors = []
with ThreadPoolExecutor(max_workers=min(len(target_student_ids), 5)) as pool:
    futures = {
        pool.submit(_gen_variant_for_student, sid, per_student_segments[sid], students_by_id): sid
        for sid in target_student_ids
    }
    for fut in as_completed(futures):
        sid = futures[fut]
        try:
            variants.append(fut.result())
        except Exception as e:
            errors.append({'student_id': sid, 'error': str(e)})

if errors:
    _logger.exception(
        "remediation.personalize.partial_failure teacher=%s class=%s standard=%s errors=%s",
        g.teacher_id, class_id, standard_code, errors,
    )
    return jsonify({
        "error": "One or more personalized variants failed to generate",
        "detail": "Try again. Personalized mode requires all variants to succeed.",
        "failed_student_ids": [e['student_id'] for e in errors],
    }), 500

# Deterministic ordering: sort by student name, fallback to id.
variants.sort(key=lambda v: (v.get('student_name') or '', v.get('student_id') or ''))

# Aggregate cost from all variants and record ONCE in the parent thread
# (Codex round 2 MINOR — _record_planner_cost is unsafe under concurrent
# writers; aggregating here avoids the lock contention).
total_usage = {}
for v in variants:
    total_usage = _merge_usage(total_usage, v.pop('usage', None) or {})
_record_planner_cost(total_usage)
```

### Backend — new batch publish endpoint

**File:** `backend/routes/student_account_routes.py`. New route alongside `publish_to_class`.

```
POST /api/publish-to-class-batch
```

Decorators: `@require_teacher`, `@handle_route_errors`. Body shape:

```json
{
  "class_id": "...",
  "items": [
    {"content": {...}, "target_student_ids": [sid], "settings": {...}, "title": "Remediation: ..."},
    {"content": {...}, "target_student_ids": [sid], "settings": {...}, "title": "Remediation: ..."}
  ],
  "content_type": "assessment"
}
```

Validation order:
1. Class ownership (`classes.teacher_id == g.teacher_id`) → 403.
2. **No duplicate target_student_ids across the batch (Codex round 2 MAJOR).** Aggregate every student_id appearing in any item's `target_student_ids`. If any student appears more than once across items, return 400 ("Duplicate target_student_id across batch items"). Defense-in-depth — by construction the personalized-mode caller never sends duplicates, but a direct API call could create more rows than the cap allows in a single request.
3. Each item validates target_student_ids: list of one (or more) UUIDs, all enrolled, all exist in students table. Same checks the existing publish_to_class does.
4. Per-student weekly cap (Phase 4.2 #8): for each unique student_id across the batch, count `existing_count` via `_check_remediation_cap` AND `pending_count` (from this batch — typically 1 per student given duplicate rejection). If `existing_count + pending_count > REMEDIATION_PER_STUDENT_WEEKLY_CAP`, the student is capped. Return 422 with `capped_student_ids`.
5. Atomic write: `db.table('published_content').insert([row1, row2, ...]).execute()` — a single PostgREST request, which PostgREST wraps in a single PostgreSQL transaction by default. The implementer MUST verify this atomicity (read supabase-py + PostgREST docs) before merging. If verification fails (extremely unlikely), STOP and bring scope back for spec revision — DO NOT fall back to compensating-delete rollback (Codex round 2 MAJOR — best-effort rollback is not equivalent to transactionality).
6. Response: `{batch_id: <uuid>, content_ids: [<uuid>, ...]}`.

The existing `/api/publish-to-class` endpoint is UNCHANGED. The batch endpoint reuses validation helpers (target_student_ids checks, cap check) but does NOT inline-import from /publish-to-class — extract any shared validation into a module-level helper if needed.

### Frontend — `RemediationDrawer.jsx`

State changes:
- New state: `mode` (string: "shared" | "personalized"), `variants` (array), `activeVariantIndex` (int, default 0).
- When response.mode === "personalized": render tab strip at top of preview pane (one tab per `variant.student_name`); preview pane shows `variants[activeVariantIndex].questions` + `variants[activeVariantIndex].lesson`.
- When response.mode === "shared": existing single-preview rendering, unchanged.

Publish flow:
- Shared mode: existing single `publishToClass(...)` call (unchanged).
- Personalized mode: NEW single `publishToClassBatch(...)` call with `items: variants.map(v => ({content: {questions: v.questions, lesson: v.lesson}, target_student_ids: [v.student_id], settings: {target_standard: standardCode}, title: "Remediation: " + standardCode}))`.

**Existing-guard branching (Codex round 2 MAJOR).** The current drawer's `publish()` checks `data.target_student_ids` and derives `nTargets` from it — these checks are SHARED-mode-specific and would refuse to publish in personalized mode (where the response has `variants[]` instead). Required branching:

- `if (mode === "personalized")`: derive `nTargets` from `variants.length`; bypass the `data.target_student_ids` guard; loop `validateBeforePublish()` over each variant's questions (a single failure → set tab-specific validation error + switch active tab to the failing variant); call `publishToClassBatch`.
- `else`: existing shared-mode flow unchanged.

Publish-button label updates accordingly: "Publish to {nTargets}" works for both modes.

New `publishToClassBatch(classId, items, contentType)` helper. Uses `fetchApi` (which surfaces non-ok response bodies per Phase 4.2 #8 fix).

### Edge cases

| Case | Behavior |
|---|---|
| Class-wide mode, 0 students with accommodations | Auto-decide stays on shared path. Existing behavior unchanged. |
| Class-wide mode, 1 of 5 students has accommodations | Personalize all 5 (uniform N). 4 prompts have empty accommodation_segment, 1 has the segment. |
| Class-wide mode, 11 students all with accommodations | 422 "Class has too many students for personalized mode". Teacher must split into single-student publishes or remove accommodations. |
| Class-wide mode, 11 students 0 with accommodations | NOT 422 — auto-decide stays shared (no cap on shared mode). |
| Class-wide mode, 11 students 1 with accommodations | 422 (>REMEDIATION_PERSONALIZED_MAX with personalize triggered). |
| Single-student mode | UNCHANGED. Single accommodation_segment, single response, single publish. Personalization branch is class-wide-only. |
| 1 of N AI calls fails | Whole /remediate call returns 500 with `failed_student_ids`. No state mutations (just LLM calls failed). Teacher retries. |
| 1 of N students is at weekly cap (Phase 4.2 #8) | Cap check at /remediate returns 422 BEFORE generation (existing behavior). Cap check at batch publish ALSO returns 422 if state changed mid-flight. |
| Drawer abandons mid-tabs (closes drawer before publish) | No state written; nothing to clean up. Generation cost was spent but that's accepted. |
| Drawer publishes successfully | N rows in `published_content`; effectiveness dashboard renders N cards; recall is per-row; weekly cap incremented per student. |
| Same standard, same teacher, two simultaneous /remediate calls | Acceptable race per Phase 4.2 #8 precedent. Both may pass cap check; both batch publishes may write. |
| `build_accommodation_prompt` raises | Caught + logged + treated as empty-string for that student. Doesn't crash the publish. |
| `build_accommodation_prompt` returns delimiter-only (no real content) | Triggers personalization (Codex MINOR caveat). Acceptable false positive. |
| Variant student name collision (two students same name) | Tab labels duplicate; sorted by name then id, so order is deterministic. UX nuance — defer; teacher can disambiguate by hovering for ID. |
| Backend response missing `mode` field (legacy /remediate response cached client-side) | Frontend defaults to "shared" mode if `mode` is undefined. Backwards-compat. |
| Batch publish at the supabase layer: `.insert(list)` semantics | Verify in implementation that `.insert([row1, row2, ...]).execute()` is atomic — single PostgREST request wrapped in a single PostgreSQL transaction. **NO compensating-delete fallback** — best-effort rollback is not equivalent to transactionality (Codex round 2 MAJOR). If atomicity verification fails, STOP and revisit scope. |
| Direct-API caller submits a batch with the same student in two items | 400 "Duplicate target_student_id across batch items" (Codex round 2 MAJOR — defense-in-depth so a malicious or buggy caller can't sneak past the cap by splitting one student across multiple items). |
| Cost telemetry under thread parallelism | Workers RETURN `usage` with their result; parent thread aggregates via `_merge_usage` and records via `_record_planner_cost` ONCE after all variants complete (Codex round 2 MINOR — `_record_planner_cost` is read-modify-write JSON without locking). |

## Out of scope (deferred)

- Async refactor of OpenAIAdapter. Threading is good enough; full async is a different project.
- DB-side transactional cap enforcement. Race condition unchanged from Phase 4.2 #8 spec.
- Per-feature lesson-token cost telemetry. Existing aggregate cost is sufficient for this slice.
- Configurable `REMEDIATION_PERSONALIZED_MAX` env var. Hardcoded at 10 unless district admins ask.
- Cross-class personalization (e.g., personalizing the same standard remediation for the same student in cls-1 and cls-2 with different prompts). Out of scope; per-class only.
- Anti-copy variant differentiation for students WITHOUT accommodations. Currently all students in personalized mode get unique AI generations (uniform N), but the prompt is identical for the no-accommodation cases — meaning their variants might be similar/identical. Genuine anti-copy diversification is a separate ask.
- "Recall all variants from this batch" UI. Each row is independently recallable; bulk recall not asked for.
- "Edit variant before publishing" inline editor. Existing drawer's question-edit UX applies per-tab; no special bulk-edit needed.
- Telemetry on personalized vs shared mode usage. Existing structured logs cover this.
