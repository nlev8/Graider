# Phase 4.2 #1 — Lesson-text Remediation

**Status:** Yellow-tier item from `project_phase4.2_backlog.md` #1.
**Differentiator vs Progress Learning:** PL ships practice with mini-lessons in front of it. Graider's Phase 4 MVP ships practice-only. This slice closes that gap with a 300-400 word AI-generated lesson (intro + worked example + key takeaway) rendered at the top of the assessment when the student opens the remediation.

## Goal

When a teacher publishes a remediation, the AI now generates a short lesson alongside the practice questions. Students see the lesson at the top of the assessment when they open it: a one-paragraph intro to the standard, one fully worked example, and a 1-2 sentence key takeaway. KaTeX-rendered math; no images. Optional reading — students who already know can scroll past instantly. The grade still comes from the practice questions only.

## Locked decisions (from brainstorming, 2026-04-29)

- **Q1 → B.** Length: ~300-400 words total. Intro paragraph + 1 fully worked example (problem + steps + final answer) + 1-2 sentence key takeaway. NOT 150-word prose-only (too thin to differentiate from "go re-read your notes"); NOT a multi-section textbook chapter (overengineering).
- **Q2 → A.** Storage: `lesson` field at top of the existing `content` JSONB on `published_content`. Specifically: `content = {questions: [...], lesson: {intro, worked_example, key_takeaway}}`. **NOT** `content.sections` — the existing `post_remediate` route flattens sections back to a flat `questions` list before publish (Codex pre-spec review caught this — early spec drafts incorrectly assumed `content.sections` existed at publish time). No schema migration. Backwards compat: existing rows have no `lesson` key → frontend renders nothing.
- **Q3 → B.** Inline at the top of the student assessment view. Specifically: rendered inside `QuestionPlayer.jsx` (which owns the question-by-question carousel) at `currentIndex === 0`, above the first question's prompt. NOT a modal pre-step; NOT a separate "Read the lesson" button. Optional read.
- **Q4 → A.** Reading is purely optional. No timer, no scroll-pin, no "I read this" checkbox. Practice questions remain the gating mechanism for the standard. The lesson does not affect the grade.
- **Q5 → A.** Single AI call. Extend the existing `post_remediate` prompt + JSON shape. ~+300-400 output tokens per remediation; same single round-trip latency. NOT two parallel calls (no coherence benefit), NOT two sequential calls (doubled latency).
- **Q6 → A.** Plain text + KaTeX math (already in the bundle). NO images. KaTeX delimiters MUST be `\(...\)` for inline and `\[...\]` for display — `renderLatex.js:6-13` (the existing helper) explicitly does NOT support `$...$` / `$$...$$`. Codex pre-spec review caught this assumption error.
- **Q7 → A.** No retro-fill. Existing remediation rows stay as-is. Frontend handles missing `lesson` key gracefully.

## Architecture

### Backend — extend `post_remediate` route

**File:** `backend/routes/student_portal_routes.py` (line ~2150 — existing route).

**Prompt extension** (replaces the `base_prompt` formulation):

The new prompt requires the AI to return JSON of this exact shape:

```json
{
  "title": "Practice - <standard>",
  "lesson": {
    "intro": "<~80-120 word paragraph explaining what the standard is and why it matters>",
    "worked_example": "<~150-200 word fully worked example: problem statement, step-by-step reasoning, final answer>",
    "key_takeaway": "<1-2 sentence summary of the rule or principle>"
  },
  "sections": [{"name": "Practice", "questions": [...]}]
}
```

The prompt MUST instruct the AI:
- Use `\(...\)` for inline math and `\[...\]` for display math (NOT dollar signs — the renderer doesn't support them).
- Total lesson length 300-400 words across the three fields.
- The worked example must demonstrate the same concept the practice questions test (within-prompt coherence).

**Accommodation-aware (single-student mode only).** The existing `accommodation_segment` (line ~2328 of `student_portal_routes.py`) currently extends the prompt for single-student mode. Append one sentence:
> "Apply these accommodations to the lesson text as well as the questions."

This is best-effort. AI compliance isn't contracted; mixed-language output (questions ELL-translated, lesson not) is acceptable degradation matching Phase 4's existing posture.

**Validation.** After the AI response and `_post_process_assignment` runs, validate the lesson:

```python
def _validate_and_clean_lesson(raw):
    """Returns a clean lesson dict {intro, worked_example, key_takeaway}
    if valid, else None. Drops unknown keys. Logs failure reason."""
    if not isinstance(raw, dict):
        _logger.warning("remediation.lesson_invalid_dropped reason=not_dict ...")
        return None
    cleaned = {}
    for field in ('intro', 'worked_example', 'key_takeaway'):
        val = raw.get(field)
        if not isinstance(val, str):
            _logger.warning("remediation.lesson_invalid_dropped reason=missing_or_nonstring_%s ...", field)
            return None
        stripped = val.strip()
        if not stripped:
            _logger.warning("remediation.lesson_invalid_dropped reason=empty_%s ...", field)
            return None
        if len(stripped) > 1500:
            _logger.warning("remediation.lesson_oversize_dropped reason=%s_%dchars ...", field, len(stripped))
            return None
        cleaned[field] = stripped
    return cleaned
```

Build the clean dict explicitly — do NOT pass-through unknown keys. Codex pre-spec caught this leak risk: AI could dump arbitrary JSON into the `published_content` JSONB if extras passed through.

**Fallback wrapper.** Today's route has a fallback at line ~2376:

```python
if isinstance(assignment, dict) and 'sections' not in assignment and 'questions' in assignment:
    assignment = {
        'title': assignment.get('title', f"Practice - {standard_code}"),
        'sections': [{'name': 'Practice', 'questions': assignment.get('questions') or []}],
    }
```

Update this wrapper to preserve `lesson`:

```python
if isinstance(assignment, dict) and 'sections' not in assignment and 'questions' in assignment:
    assignment = {
        'title': assignment.get('title', f"Practice - {standard_code}"),
        'lesson': assignment.get('lesson'),  # NEW
        'sections': [{'name': 'Practice', 'questions': assignment.get('questions') or []}],
    }
```

Otherwise the wrapper drops lesson when AI returns `{questions, lesson}` shape instead of `{sections, lesson}`.

**Response shape.** The route ALWAYS returns `lesson` in the response body. Value is the clean dict if validation passed, `None` if it didn't. Frontend null-check is consistent.

```python
return jsonify({
    "questions": questions,
    "lesson": clean_lesson,  # dict or None
    "target_mode": target_mode,
    "target_student_ids": target_student_ids,
    "standard_code": standard_code,
    "generated_at": ...,
}), 200
```

### Backend — RemediationDrawer publish payload (no backend route change, but contract change)

The drawer's publish payload (sent to `POST /api/publish-to-class`) currently sends `content: {questions: [...]}` (verified at `RemediationDrawer.jsx:187`). Update the drawer to include lesson:

```javascript
content: { questions: questions, lesson: lesson }
```

`publish_to_class` (in `student_account_routes.py:474`) writes `content` verbatim to JSONB — no backend route change needed. Codex pre-spec caught that without this drawer change, lesson would silently disappear before storage.

### Frontend

**`frontend/src/tabs/RemediationDrawer.jsx`** (modified, ~+30 LOC):
- Receive `lesson` from `/remediate` response. Store in state alongside `questions`.
- Render lesson preview block above the questions preview, using the same `<LessonBlock>` component the student-side will use.
- Pass `lesson` in the `content` object to `publishToClass(...)`.

**`frontend/src/components/StudentPortal.jsx`** (modified, +2 LOC):
- Pass `lesson={assessment?.content?.lesson}` prop to `<QuestionPlayer />` (line ~544).

**`frontend/src/components/QuestionPlayer.jsx`** (modified, ~+15 LOC):
- Accept `lesson` prop.
- When `currentIndex === 0` AND `lesson` is a non-null dict, render `<LessonBlock lesson={lesson} />` ABOVE the question prompt. On subsequent questions, hide the lesson — students who reached question 2+ have already seen it.

**New file: `frontend/src/components/LessonBlock.jsx`** (~80 LOC):
- Three labeled sections rendered in order: Intro, Worked Example, Key Takeaway.
- Each section's prose runs through the existing `renderQuestionText` helper (the wrapper around `renderLatex` used elsewhere) so KaTeX `\(...\)` / `\[...\]` math renders. Direct `renderLatex` is acceptable if `renderQuestionText` doesn't fit.
- Card chrome matches existing question-card styling (uses `glass-card` class).
- No markdown library; the lesson fields are plain text + KaTeX. Line breaks via `\n` inside the prose render as paragraph breaks via `String.fromCharCode(10)`-aware splitting (CLAUDE.md prohibits multi-line string literals; use the existing pattern).

### Edge cases

| Case | Behavior |
|---|---|
| AI returns no `lesson` key | Log `remediation.lesson_invalid_dropped reason=not_dict`. Response has `lesson: null`. Frontend renders nothing. |
| AI returns `lesson` as wrong type (string, list, null) | Same `reason=not_dict`. Drop. |
| AI returns `lesson` dict missing per-field | `reason=missing_or_nonstring_<field>`. Drop. |
| AI returns `lesson` per-field empty/whitespace | `reason=empty_<field>`. Drop. |
| AI returns `lesson` per-field non-string (e.g., `key_takeaway: 42`) | `reason=missing_or_nonstring_<field>`. Drop. |
| AI returns `lesson` per-field over 1500 chars | `reason=<field>_<len>chars`. Drop. (Don't truncate — could cut mid-equation.) |
| AI returns extra fields in `lesson` (e.g., `summary`, `tags`) | Validator builds clean dict with ONLY the three known fields. Extras are dropped before publish. NO unbounded data leak into JSONB. |
| Malformed KaTeX in prose | `renderLatex.js` falls back to plain text. No additional handling. |
| Existing legacy remediation row (no `lesson` key) | Drawer + student renderer handle missing key gracefully (render nothing). |
| `publish_to_class` content reshape | Verified pass-through — writes `content` JSONB verbatim from request body. No backend route change needed. |
| Student loads recalled (`is_active=false`) remediation | Phase 4.2 #5 visibility helper blocks before content served. Lesson never reaches client. |
| Per-student differentiation: AI ignores accommodation in lesson | Best-effort; documented; matches Phase 4 question-side posture. |
| Within-call coherence: lesson worked-example contradicts a question | Not validated. AI's job. Documented as accepted risk. |
| AI generates valid lesson but questions fail (<3 valid) | Existing 422 floor wins. Whole request 422; lesson discarded. |
| XSS / HTML injection in lesson prose | Prose renders as JSX text children (auto-escape). KaTeX's `dangerouslySetInnerHTML` only renders KaTeX's own output. No sanitization library. |
| Combined lesson size | Per-field 1500 chars × 3 = ≤4500 chars total. JSONB writes/reads stay performant. |
| Teacher wants to edit lesson before publishing | NOT supported in this slice. Re-click cell to regenerate. Documented out-of-scope. |
| Math delimiter mismatch (`$...$` from older AI runs or copy-paste) | Renders as plain text via `renderLatex.js` fallback. Not visually broken, just unrendered. |
| Drawer fails to fetch / parse `/remediate` response | Existing Phase 4 drawer error UI applies. Lesson absence is one of many failure modes the existing UI already handles. |

### Compliance

## Out of scope (deferred)

- Teacher inline edit of lesson before publishing. If teacher wants to fix a typo or factual error, they regenerate by re-clicking the red cell. Inline editing pulls in textarea state + validation re-run + likely a markdown editor dependency. Revisit when teacher feedback names the friction.
- Retroactive backfill of existing remediations. AI cost on already-engaged content; backfill could surprise students mid-flight. Existing rows stay as-is.
- Lesson-only content type (`study_guide` row published standalone). The Phase 4.2 backlog item #1 mentioned this as an option — out of scope for this slice (would require a separate publish flow). Could ship later as a sibling feature.
- Image-based worked examples (diagrams, charts). Phase 5c image-adapter infra exists but adds cost + safety + storage. Not justified for MVP.
- Multi-section / multi-example lessons (long form). 1 worked example is enough for the smallest meaningful slice.
- Student-side "highlight / note" interaction with the lesson. Read-only consumption.
- Telemetry on "did the student actually read the lesson" (scroll depth, time-on-page). Optional read by design (Q4 → A).
- Per-feature lesson-token cost telemetry. Existing `feature_label="remediation"` rolls up the total cost; if product wants the lesson share isolated, do it as a follow-up.
- Math delimiter improvement for question prompts. Codex review noted that `QuestionPlayer.jsx` currently renders `q.question` as plain JSX with no `renderLatex` call — questions today don't get math rendering either. Wiring `renderLatex` into question prompts is a worthwhile follow-up but out of scope here. This slice fixes the lesson side only.
