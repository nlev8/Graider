# Phase 4.5 — Per-User JWT Supabase Clients

**Status:** Draft — awaiting Codex cross-check + user approval
**Authors:** Alexander Crionas + Claude Opus 4.7
**Date:** 2026-04-19
**Depends on:** PR #102 (Alembic tooling) merged; PR #103 (Phase 4.2 PR2 RLS) merged.
**Roadmap phase:** Phase 4.5 — activates the RLS shipped in 4.2 PR1 + PR2 for direct-JWT teachers.

---

## Motivation

Phase 4.2 PR1 + PR2 shipped canonical RLS policies on 14 tables. Those
policies are currently **decorative**: the backend uses the Supabase
service-role key, which bypasses RLS by design. RLS only becomes
load-bearing when requests hit Postgres with a user-scoped JWT in the
`Authorization` header — which is what Phase 4.5 introduces.

This spec does not attempt to migrate every auth surface. It scopes
specifically to **routes where the request already arrives with a
validated Supabase Bearer JWT** (paths that flow through
`backend/auth.py:208`'s `validate_token()` branch). Other auth
sources — Clever SSO, ClassLink SSO, student portal tokens, admin,
background/cron — keep using the service-role client exactly as they
do today.

**Why not all auth paths:** SSO users don't arrive with Supabase JWTs.
Minting JWTs for them requires designing a session model (provisioning,
storage, refresh, revocation) that's materially bigger than Phase 4.5.
That work is deferred to a future Phase 4.5b with its own spec.

---

## Goals and non-goals

### Goals

- Direct-JWT teacher requests hit Postgres as `authenticated` role
  with correct `auth.uid()`, so RLS policies actually enforce
  ownership
- Clever, ClassLink, student portal, admin, dev-mode, background,
  and webhook paths continue using service-role (no regression)
- Traffic split is observable: log every API request with `db_mode`
  (service_role | per_user_jwt) and `auth_source` (jwt | clever |
  classlink | student | dev | background | none)
- Instrumentation ships BEFORE the behavior flip, so we measure
  direct-JWT traffic share in production before committing to the
  change
- Rollback is a single env var flip + rolling restart; no data
  migration, no code revert

### Non-goals

- **SSO unification.** Deferred to Phase 4.5b. Cannot be done as a
  "small extension" — see § Design decisions § Option B rejected.
- **Student portal RLS.** Separate policy surface; out of Phase 4.5
  scope.
- **Admin escape hatch redesign.** Admin routes legitimately need
  cross-teacher access; they keep using the service-role helper.
- **Client caching.** Fresh client per request. Revisit only if
  latency data at p95 proves it's warranted.
- **Dual-mode fallback.** Failures under per-user mode fail loudly,
  not silently. "Empty results" is not a reliable RLS-breakage
  signal — legitimate empty responses would be masked and actual
  policy gaps would be hidden.

## Design decisions (brainstorm outcomes)

### Scope: A — direct-JWT only

Three scope options were considered:

| Option | Scope | Verdict |
|---|---|---|
| A | Direct-JWT teachers only | **Selected** |
| B | All teacher paths (A + SSO JWT-minting) | Rejected |
| C | Big-bang including student portal | Rejected |

**Why B was rejected:** Supabase's admin APIs (`auth.admin.generate_link()`,
`auth.admin.create_user()`) do not produce ready-to-use server-side session
JWTs. `generate_link()` produces email/OTP sign-in material a user clicks;
`create_user()` creates the user row but issues no tokens. Supabase's
documented session model requires the user to actually sign in via some
flow (password, OTP, OAuth) that returns an access+refresh token pair.
Getting Clever/ClassLink users into that flow is a session-architecture
redesign (provisioning rules, token storage, refresh semantics, revocation,
mid-request failure handling), not a moderate extension of A. An earlier
draft of this spec claimed the SSO JWT-minting was "straightforward" and
referenced these admin APIs — the claim was wrong and is corrected here.

**Why C was rejected:** Student portal runs on a custom token system
(`X-Student-Token` header, sessions stored in `student_sessions` table).
It has no RLS policy surface in 4.2 PR1/PR2 (those policies are
teacher-scoped, not student-scoped). Forcing student RLS into 4.5 bundles
a new policy design project into what should be a database-access-mode
change.

### Rollout: B-prime — new accessor + migration + global flag

Three rollout options were considered:

| Option | Pattern | Verdict |
|---|---|---|
| A | Per-route feature flags | Rejected (operator overhead, combinatorial test matrix) |
| B (original) | Change `get_supabase()` behavior under a global flag | Rejected |
| B-prime | New `get_request_supabase()` accessor + explicit call-site migration + global flag | **Selected** |
| C | Dual-mode with auto-fallback on RLS error | Rejected |

**Why B (original) was rejected:** the existing `get_supabase()` helper
at `backend/supabase_client.py:51` is called from all auth surfaces —
Clever (`backend/routes/classlink_routes.py:66`,
`backend/routes/clever_routes.py:45`), admin
(`backend/routes/admin_routes.py:218`), student portal
(`backend/routes/student_portal_routes.py:55`), student account
(`backend/routes/student_account_routes.py:57`), behavior
(`backend/routes/behavior_routes.py:28`), and background
(`backend/services/portal_grading.py:411`). Changing that helper's
decision logic to "per-user if JWT present" would affect non-JWT paths
too; even if the outcome is "service-role because no JWT," the edge
cases (background tasks executed inside request contexts, race
conditions across threads, accidental JWT bleed) are unacceptable for
a path where silent RLS-bypass = data leak.

**Why B-prime is correct:** adding a NEW accessor `get_request_supabase()`
and migrating direct-JWT callers to it explicitly produces:
- A call-site audit trail (grep for the new accessor to enumerate
  RLS-under routes)
- Zero behavior change for non-migrated callers (they still use
  `get_supabase()` which is unchanged)
- Safe incremental migration (flag OFF ⇒ both accessors return
  service-role; flag ON ⇒ only `get_request_supabase()` returns
  per-user when a JWT is present)

**Why C was rejected:** "Empty results" is not a reliable signal for
RLS breakage — empty is often a legitimate business outcome. Silent
fallback to service-role when a per-user query returns unexpected
empties would hide policy gaps exactly when we most need to surface
them.

### Client construction: B (corrected) — fresh per-request client with anon key + Authorization header override

Three client-construction approaches were considered:

| Option | Approach | Verdict |
|---|---|---|
| A | `postgrest.auth(jwt)` on shared singleton | Rejected (thread-unsafe) |
| B (original) | Fresh `create_client(URL, user_jwt, options)` | Rejected (wrong slot — user JWT isn't a Supabase API key) |
| B (corrected) | Fresh `create_client(URL, anon_key, options={headers: {"Authorization": "Bearer <jwt>"}})` | **Selected** |
| C | Cached per-(user, jwt_sig) client | Rejected (cache invalidation complexity) |

**Why B (original) was wrong:** passing the user JWT as the second
positional arg to `create_client()` puts it in the `apiKey` header,
which must be a Supabase project key (anon or service). A user JWT
there produces a malformed request PostgREST will reject. The second
arg must stay as a project key; the user JWT goes in `Authorization`
via `ClientOptions.headers`.

**Why anon key (not service key) in the `apiKey` slot:** defense in
depth. PostgREST uses both headers: `apiKey` identifies the project,
`Authorization` identifies the user. When both are present, the
`Authorization` JWT's role is authoritative for RLS. But if the
`Authorization` header is ever dropped mid-flight (middleware bug,
httpx transport issue, refactor regression), PostgREST falls back to
interpreting the `apiKey` alone. Service key → `service_role` →
bypasses RLS → silent data leak. Anon key → `anon` role → no access
to teacher tables → visible error, not silent leak. Fail closed.

Cost: one new env var `SUPABASE_ANON_KEY` (Railway + CI + `.env.example`).
The anon key is already hardcoded in public frontend assets (`landing/
script.js:6-8`, `backend/static/assets/index-*.js:344`) — safe to also
expose to the backend env. Supabase considers anon keys public and
RLS-protected.

**Why C was rejected:** JWT expiry is 1 hour. Caching clients keyed
by JWT signature means holding references to potentially-stale auth
for hours. The invalidation logic is subtle; the bug it produces
("my reads return empty after 1 hour!") is the exact "RLS silently
broken" hazard we're trying to eliminate. Revisit only if measured
per-request `create_client()` latency exceeds 20ms at p95.

## Implementation

### `backend/supabase_client_scoped.py` — the new accessor

```python
"""Request-scoped Supabase client accessor for Phase 4.5.

Returns a per-user client when all three conditions hold:
  1. USE_PER_USER_JWT env var is "1"
  2. The current Flask request has a validated Supabase Bearer JWT
     (set on g.supabase_jwt by backend/auth.py's check_auth hook)
  3. SUPABASE_URL and SUPABASE_ANON_KEY are configured

Otherwise returns the existing service-role singleton unchanged —
guaranteeing zero behaviour change for non-migrated callers.

Do NOT import this from code paths that don't execute in a Flask
request context (Celery tasks, cron jobs, background threads). Those
must continue importing from backend.supabase_client directly.
"""
from __future__ import annotations

import os

from flask import g
from supabase import create_client
from supabase.lib.client_options import ClientOptions

from backend.supabase_client import get_supabase as _get_service_client
from backend.supabase_resilient import ResilientClient


def get_request_supabase():
    """Return the right Supabase client for the current request.

    See module docstring for decision rules.
    """
    if os.getenv("USE_PER_USER_JWT", "0") != "1":
        return _get_service_client()

    user_jwt = getattr(g, "supabase_jwt", None)
    if not user_jwt:
        # SSO session, student token, background — all return None here.
        return _get_service_client()

    url = os.getenv("SUPABASE_URL")
    anon_key = os.getenv("SUPABASE_ANON_KEY")
    if not url or not anon_key:
        raise RuntimeError(
            "USE_PER_USER_JWT=1 requires SUPABASE_URL + SUPABASE_ANON_KEY "
            "to be configured. Set them on Railway before flipping the flag."
        )

    raw = create_client(
        url,
        anon_key,
        options=ClientOptions(
            auto_refresh_token=False,
            persist_session=False,
            headers={"Authorization": f"Bearer {user_jwt}"},
        ),
    )
    return ResilientClient(raw)
```

### `backend/auth.py` — set `g.supabase_jwt`

One line added to the existing Bearer-token validation branch in
`check_auth()` (currently around line 224):

```python
# EXISTING CODE (unchanged context shown for precision):
token = auth_header[7:]  # Strip 'Bearer '
payload = validate_token(token)
if payload is None:
    return jsonify({'error': 'Invalid or expired token'}), 401

# Attach user info to Flask's g object for use in route handlers
g.user_id = payload.get('sub')
g.user_email = payload.get('email', '')
g.supabase_jwt = token   # NEW — only set on direct-JWT path
```

`g.supabase_jwt` is NEVER set on Clever / ClassLink / dev-header
paths (those branches return earlier in `check_auth()` and don't
reach this line). `get_request_supabase()` treats its absence as
"service-role."

### `backend/observability/db_mode.py` — structured logging

```python
"""Emit a structured log line per API request describing the effective
database access mode and authenticated identity source. Used to
measure traffic splits before and after the USE_PER_USER_JWT flag flip.
"""
from __future__ import annotations

import logging
import os

from flask import Flask, g, request, session

_logger = logging.getLogger("backend.db_mode")


def _classify_auth_source() -> str:
    """Return one of: jwt | clever | classlink | student | dev | none.
    Mirrors the branch order in backend/auth.py's check_auth hook.
    """
    if getattr(g, "supabase_jwt", None):
        return "jwt"
    if "clever_user" in session:
        return "clever"
    if "classlink_user" in session:
        return "classlink"
    if request.headers.get("X-Student-Token"):
        return "student"
    if request.headers.get("X-Test-Teacher-Id"):
        return "dev"
    return "none"


def _classify_db_mode() -> str:
    """Return service_role | per_user_jwt.
    Mirrors the decision in get_request_supabase().
    """
    if os.getenv("USE_PER_USER_JWT", "0") != "1":
        return "service_role"
    if getattr(g, "supabase_jwt", None):
        return "per_user_jwt"
    return "service_role"


def register(app: Flask) -> None:
    """Register the after_request logger. Called from backend/app.py."""

    @app.after_request
    def _log_db_mode(response):
        if request.path.startswith("/api/"):
            _logger.info(
                "request.db_mode",
                extra={
                    "path": request.path,
                    "method": request.method,
                    "status": response.status_code,
                    "auth_source": _classify_auth_source(),
                    "db_mode": _classify_db_mode(),
                    "user_id": getattr(g, "user_id", None),
                },
            )
        return response
```

Registered in `backend/app.py` alongside the existing extension hooks.

## Call-site migration — tiered

Direct-JWT routes are a subset of ~25 handlers. Migration happens in
small PRs grouped by risk tier:

### Tier 1 — low-risk reads (migrate first)

Routes that mostly read small data surfaces. Any RLS gap surfaces here
with minimal blast radius.

- `backend/routes/settings_routes.py` (teacher settings, rubric,
  presets)
- `backend/routes/survey_routes.py` (survey creation + results)
- `backend/routes/assessment_results_routes.py`
- `backend/routes/analytics_routes.py`
- `backend/routes/assistant_routes.py` (assistant chat history reads)

### Tier 2 — grading + publish (migrate after Tier 1 validates plumbing)

Routes that touch the richest data paths.

- `backend/routes/planner_routes.py`
- `backend/routes/grading_routes.py`
- `backend/routes/assignment_routes.py`
- `backend/routes/assignment_player_routes.py`
- `backend/routes/student_portal_routes.py` (teacher-publish side
  only; student-read side keeps service-role — student portal out of
  4.5 scope)
- `backend/routes/lesson_routes.py`
- `backend/routes/behavior_routes.py` (teacher-authored behavior
  data)
- `backend/routes/document_routes.py`
- `backend/routes/email_routes.py`
- `backend/routes/automation_routes.py`
- `backend/routes/seo_routes.py`
- `backend/routes/assessment_player_routes.py`

### Tier 3 — excluded (stay on service-role by policy)

- `backend/routes/admin_routes.py` (cross-teacher admin access)
- `backend/routes/clever_routes.py` (SSO — not direct-JWT)
- `backend/routes/classlink_routes.py` (SSO)

- `backend/routes/student_account_routes.py` (student-side reads)
- `backend/routes/sync_routes.py` (webhook auth, not JWT)
- `backend/routes/stripe_routes.py` (webhook, service-role correct)
- `backend/routes/lti_routes.py` (LTI platform integration)
- `backend/routes/district_routes.py` (session-based admin)
- `backend/routes/auth_routes.py` (admin-API-heavy; stays service-role)

**Migration mechanics per file:** replace `from backend.supabase_client
import get_supabase` with `from backend.supabase_client_scoped import
get_request_supabase`, then search-and-replace `get_supabase(` →
`get_request_supabase(` within the file. No logic changes. Each PR
runs the full test suite; Migrations Smoke CI covers RLS policy
compatibility; destructive-op scanner doesn't apply (no Alembic
changes).

## Testing

### Unit tests (`tests/test_supabase_client_scoped.py`, new)

- `get_request_supabase()` with flag off → returns the service-role
  singleton (mock `_get_service_client()`, assert returned object
  is the mock)
- Flag ON + no `g.supabase_jwt` → service-role
- Flag ON + `g.supabase_jwt` set + env vars present → calls
  `create_client(SUPABASE_URL, SUPABASE_ANON_KEY, options=…)` with
  `Authorization: Bearer <jwt>` in options.headers and returns a
  `ResilientClient` wrapping the result
- Flag ON + JWT + missing env vars → raises `RuntimeError` with
  diagnostic message
- Tests live under a small Flask `app_context` + `request_context`
  helper to make `g` + `request` available

### Integration test (`tests/test_per_user_rls_enforcement.py`, new)

Proves the end-to-end plumbing actually enforces RLS. Uses the same
ephemeral-Postgres pattern as Phase 4.2 PR2's Migrations Smoke:

1. Spin up `postgres:15-alpine` service container with the
   `supabase_stubs.sql` auth schema + stub roles
2. Apply PR1 + PR2 migrations (seed the RLS policies)
3. Seed two teachers and two rows each in a policy-gated table
   (e.g. `classes`)
4. Mint two JWTs using `SUPABASE_JWT_SECRET` + `jwt.encode()` with
   `sub=<teacher_id>`, `aud=authenticated`, `role=authenticated`,
   `exp=now + 1h`
5. Run request as teacher A with A's JWT; assert only A's class is
   visible
6. Run same request as teacher B with B's JWT; assert only B's
   class is visible
7. Run request with no JWT (simulates service-role fallback); assert
   both classes are visible (service-role bypasses RLS)

This test is the load-bearing one — it proves RLS policies + per-user
client + accessor stitch together correctly. Gated in CI behind the
existing Migrations Smoke job.

### Regression coverage

Existing 1581+ tests must stay green with flag OFF throughout the
rollout. Phase 4.5 is strictly additive until 6e.

## Observability bar before flipping

Phase 6e (flag flip) SHALL NOT happen until:

1. **Traffic split is visible.** Structured logs show daily breakdown
   of `auth_source` (jwt vs clever vs classlink vs student vs none)
   and `db_mode`. Baseline captured for ≥2 days.
2. **Zero 5xx from the new code path.** No requests have hit the
   "missing SUPABASE_ANON_KEY" RuntimeError in `get_request_supabase()`.
3. **PostgREST permission-error rate baseline is 0.** Before the
   flip, per-user mode is off, so no `42501` errors should appear.
   Post-flip we'll measure the delta.
4. **Per-request `create_client()` latency measured ≤20ms at p95.**
   If it's higher, we revisit cache strategy before flipping.

Post-flip monitoring dashboards watch:
- 401/403 spike (JWT validation regression — shouldn't happen since
  validation is unchanged)
- PostgREST `42501` errors (RLS policy gap — forward-fix via new
  migration)
- Latency regression at p95 on direct-JWT routes
- Canary-endpoint empty-result anomalies (handful of known-non-empty
  queries per-route, not a blanket "empty=bad" signal)

---

## Recovery model

All failures post-flip are forward-fix:

- **PostgREST permission error on a direct-JWT request:** check RLS
  policy coverage for the table; if policy is missing or restrictive,
  write a new Alembic revision that adds/relaxes the policy. Never
  revert code to service-role — that defeats the whole phase.
- **Unexpected latency at p95:** flip the flag back to `0` if
  severe; otherwise tune (consider client caching from the deferred
  list).
- **Widespread breakage:** flip `USE_PER_USER_JWT=0`, rolling restart.
  Service-role behavior restored instantly. Post-mortem to diagnose.

---

## Explicitly deferred

- **Phase 4.5b — SSO unification.** Covers Clever + ClassLink users
  becoming real Supabase Auth users, session token lifecycle,
  refresh, revocation, linking/provisioning rules, failure modes on
  mid-request refresh. Its own brainstorm + spec; not a moderate
  extension of 4.5.
- **Student portal RLS.** Separate policy surface; student-reads-own
  semantics that PR1/PR2 didn't design.
- **Per-request client caching.** Revisit only if latency metrics
  justify the invalidation complexity.
- **Removing `get_supabase()`.** Stays indefinitely for admin, SSO,
  background, and webhook paths. `get_request_supabase()` never
  becomes the universal accessor.
- **Admin-scoped RLS reads on `audit_log` and similar.** Separate
  authorization-model spec.
