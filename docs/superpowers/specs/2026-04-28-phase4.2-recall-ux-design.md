# Phase 4.2 #5 — Remediation Recall UX

**Status:** Yellow-tier item from `project_phase4.2_backlog.md` #5.
**Differentiator vs Progress Learning:** PL has no equivalent recall affordance — teachers either delete (data loss) or wait for the assignment to expire. Graider's recall is a soft-flag flip that preserves audit data while immediately hiding the remediation from students who haven't engaged.

## Goal

Teacher publishes a remediation, immediately realizes it's wrong (wrong standard, wrong students, wrong question count), recalls it from the new effectiveness dashboard in 2 clicks, and students who haven't yet engaged stop seeing it on their dashboard. Submissions already made stay intact.

## Locked decisions (from brainstorming, 2026-04-28)

- **Q1 → A.** "Recall" semantics = flip `published_content.is_active` to false. No new schema column. No `recalled_at` audit column. Submissions preserved. The visibility helper `_content_visible_to_student` (Phase 4 Bundle 2) already gates on `is_active` — recalled remediations stop being visible to students immediately, no cache invalidation needed.
- **Q2 → A.** Recall button lives on the per-remediation card in the new effectiveness dashboard ONLY. No legacy assessments-list integration in this slice. No cross-class-content rollout — keep the surface area minimal.
- **Q3 → B.** Confirmation flow: button → modal ("Recall this remediation? N students already submitted — their work is preserved. They'll just no longer see it in their dashboard.") → confirm → flip + toast. No optimistic-flip-with-undo pattern (would require un-recall code path; out of scope).
- **Q4 → B.** Dashboard treatment: recalled remediations show a "Recalled" badge in card header + keep the data rows visible. "Remediate again" CTA stays available — teachers may want to publish a corrected version. Card is NOT grayed out; preserves the audit trail for post-mortem reading.
- **Q5 → A.** Student-already-submitted count surfaced in the modal body (informs decision) AND in the success toast. Count comes from the dashboard row data already in frontend state; zero extra fetches.

## Architecture

### Backend — one new route

**Route:** `POST /api/teacher/class/<class_id>/remediation/<rem_id>/recall` in `backend/routes/student_portal_routes.py`. Decorators: `@require_teacher`, `@handle_route_errors`. Body-less.

**Validation order:**
1. Class ownership (`classes.teacher_id == g.teacher_id`) → 403 via `error_response("Not authorized", 403)`. Same pattern as the dashboard route.
2. Remediation lookup: select `id, is_active, target_student_ids` from `published_content` where `id == rem_id` AND `class_id == class_id`. If not found → 404 via `error_response("Remediation not found", 404)`. Class-scoping in the WHERE clause is a second defense-in-depth layer beyond the ownership check.
3. Defense-in-depth: refuse if `target_student_ids IS NULL` (Python check, since the path says "remediation" — endpoint should not accept non-remediation content). 404 via `error_response("Remediation not found", 404)`. Identical message intentionally — don't leak whether the ID exists as non-remediation content.
4. Idempotent: if `is_active == false` already, return 200 with `{recalled: true, already_recalled: true, rem_id}` (no second write). Avoids surfacing a confusing "error" when the teacher's local state was stale.

**Action:**
```python
db.table('published_content').update({'is_active': False}).eq('id', rem_id).eq('class_id', class_id).execute()
```
Class-scoping in the WHERE clause is intentional belt-and-suspenders even though step 2 already verified.

**Response shape:**
```json
{
  "recalled": true,
  "already_recalled": false,
  "rem_id": "uuid"
}
```
No timestamp in the response. The toast text doesn't need one (generic message — see Frontend section), and adding a server-generated timestamp would be misleading for the `already_recalled: true` case (it would NOT reflect when the recall actually happened — Q1 → A explicitly defers a stored `recalled_at` column).

**Structured logging:** one event per successful recall:
```
remediation.recalled rem_id=<uuid> class=<uuid> teacher=<uuid> already_recalled=<bool>
```

### Backend — extend effectiveness dashboard response

In the existing `get_class_remediation_effectiveness` route, add `is_active` to each remediation card's response:

```json
{
  "remediation_id": "uuid",
  "is_active": true,
  "title": "...",
  // ... existing fields
}
```

This is a single field addition to the response builder. Frontend uses it to render the "Recalled" badge.

The dashboard query continues to fetch ALL `target_student_ids IS NOT NULL` rows regardless of `is_active` — recalled remediations stay visible in the dashboard for audit purposes (Q4 → B).

- Component signature changes from `({ classId })` to `({ classId, addToast })`. `addToast` is already in scope in `AnalyticsTab` (`AnalyticsTab.jsx:2371`); add it to the prop pass at `AnalyticsTab.jsx:2631` where `RemediationEffectiveness` is rendered.
- Add "Recall" button to each card header (right side, next to existing card title block).
- When `remediation.is_active === false`, show "Recalled" badge instead of the Recall button. Badge styled same as Phase 3a Gradebook status badges.
- Inline confirm modal component (~30 LOC, no new file). Modal text:
  > "Recall this remediation? **N students already submitted** — their work is preserved. They'll just no longer see it in their dashboard."
  >
  > Where `N` is computed from `remediation.rows.filter(r => r.completed).length`. Drop the bolded sentence entirely if N=0.
- Modal buttons: "Cancel" (closes), "Recall" (red, calls `recallRemediation` then re-fetches dashboard).
- Modal-local error state: if `recallRemediation` throws, show inline error banner inside the modal (don't close); teacher can retry. On success, close modal + emit toast.
- Success toast (via `addToast`): `"Recalled. {N} students had already submitted — their work is preserved."` (drop second sentence if N=0).

**`frontend/src/services/api.js`** (modified, +10 LOC). MUST use the existing `fetchApi()` wrapper (api.js:31) — it injects Supabase auth headers and handles 401 retry. Raw `fetch()` would 401 for normal teacher sessions:
```javascript
export async function recallRemediation(classId, remId) {
  return fetchApi(
    '/api/teacher/class/' + encodeURIComponent(classId) +
    '/remediation/' + encodeURIComponent(remId) + '/recall',
    { method: 'POST' }
  );
}
```

### Edge cases

| Case | Behavior |
|---|---|
| Teacher recalls a remediation in a class they don't own | 403 via `error_response("Not authorized", 403)`. |
| `rem_id` is not a remediation (`target_student_ids IS NULL`) | 404 via `error_response("Remediation not found", 404)`. Endpoint is remediation-specific by name. |
| `rem_id` is in a different class than the path's `class_id` | 404 (the `.eq('class_id', class_id)` filter misses). |
| Already recalled (`is_active = false`) | 200 idempotent. Body: `{recalled: true, already_recalled: true, ...}`. No second write. |
| Student submits DURING recall (race) | Submission still goes through (visibility helper read at submit time may have raced); future attempts blocked. Dashboard re-fetch reflects latest attempt count. Acceptable — submitted work always preserved. |
| Backend update fails | `@handle_route_errors` returns 500 via RFC 7807 envelope. Frontend modal stays open with error banner; teacher can retry. Optimistic state NOT applied. |
| Frontend re-fetch after recall fails | Toast shows; card shows recalled badge from optimistic state; subtle banner offers manual refresh. Recall succeeded; only the refresh failed. |
| `target_student_ids = []` (empty array) | publish_to_class rejects empty arrays at write time, so this shouldn't exist. If it does (manual DB edit), the existing dashboard `is not None` filter still includes it. The recall endpoint also accepts it (defense-in-depth check is `is None`, not `is None or len() == 0`). Acceptable — recalling something that shouldn't exist is harmless. |

### Compliance

Reads and writes only `published_content.is_active`. No schema migration. No PII added to logs. Zero touch to Clever / ClassLink / OneRoster / LTI surfaces. The recall is fully reversible at the database level — a future "Restore" feature could surface un-recall via UI without any data backfill.

## Out of scope (deferred)

- Un-recall / Restore UI. The DB column flip is reversible (just set `is_active=true` again), so a future feature can add a sibling `restore` endpoint + a "Restore" button on recalled cards without any schema or data migration. Not in this slice because Q1 → A explicitly scoped this out.
- Cross-class generic toggle. A more general `POST /api/teacher/class/<id>/content/<id>/toggle` route reusable for any class-based content type. Currently no consumer outside remediations — YAGNI per CLAUDE.md.
- `recalled_at` timestamp column on `published_content`. Would let the dashboard distinguish "recalled" from "deactivated for unrelated reasons" and timestamp-sort the audit trail. Currently `is_active=false` only happens via this route, so the inference is unambiguous. Add column when a second use case emerges.
- Bulk recall ("recall all remediations for a class"). No teacher request.
- Recall reason / note. Teachers don't currently leave notes on assessments; consistent with existing patterns to keep this feature note-less.
