# Phase 4.2 — Submit-routing fix (Design)

**Status:** Bugfix. Pre-existing routing conflict surfaced during Phase 4 implementation (PR #139, 2026-04-27).

**Goal:** Make `submit_student_work` reachable in production. Class-based authenticated student submissions actually succeed instead of 404'ing through the join-code handler. Bundle 2's visibility helper does its job.

**Tracked from:** `project_phase4.2_backlog.md` item #11.

## Background

Two blueprints register `/api/student/submit/<X>` with the same Flask URL pattern:

| Blueprint | Handler | URL pattern | Purpose |
|---|---|---|---|
| `student_portal_bp` | `submit_assessment(code)` | `/api/student/submit/<code>` | Anonymous join-code path; queries `published_assessments.join_code` |
| `student_account_bp` | `submit_student_work(content_id)` | `/api/student/submit/<content_id>` | Authenticated class-based path; queries `published_content`; visibility-checks via `_content_visible_to_student` |

Werkzeug treats both as `/api/student/submit/<string>` and silently uses the first registered rule. `student_portal_bp` is registered at `backend/routes/__init__.py:55`, `student_account_bp` at line 61 — so `submit_assessment` wins.

**Impact:**
- Authenticated class-based students POSTing to `/api/student/submit/<UUID>` actually hit `submit_assessment`. The handler queries `published_assessments.eq('join_code', UUID)`, finds nothing (UUID isn't a 6-char join code), returns `404 "Assessment not found"`.
- Confirmed by local test client repro (2026-04-27): POST `/api/student/submit/abc-12345-content-id-uuid` returns 404 with the join-code path's response shape.
- `submit_student_work` is currently **dead code in production**.
- Bundle 2's `_content_visible_to_student` application to `submit_student_work` is therefore dead.
- The corresponding Phase 4 deny-test (`test_submit_student_work_404_for_non_targeted`) was skipped with a documented `pytest.mark.skip`.
- A pre-existing comment at `tests/test_student_account_coverage.py:207-211` already acknowledged the conflict and bypassed Flask routing in tests by invoking `submit_student_work` directly via `app.test_request_context`.

## Locked decisions

- **Architecture (Q1) → B.** Migrate `submit_student_work` to a distinct URL prefix. NOT reorder blueprints (would break the active anonymous join-code path) and NOT migrate the join-code path (more frontend churn — `submitStudentAssessment` is the API client function callers use).
- **URL name (Q2) → A.** New URL is `/api/student/class-submit/<content_id>`. Symmetric with existing `class_id`-based teacher routes (`/api/teacher/class/<class_id>/...`). Reads naturally as "the class-based submit path."

## Architecture

### Backend — one route URL change + one auth-prefix add

**Route URL change.** In `backend/routes/student_account_routes.py` find the `@student_account_bp.route('/api/student/submit/<content_id>', methods=['POST'])` decorator on `submit_student_work` (line 866 at HEAD `fb01c4c`). Change the path to `/api/student/class-submit/<content_id>`. No handler-body changes — `submit_student_work` already has:
- `_validate_student_session()` (Bundle 2 added enrollment recheck + UUID validation)
- `_content_visible_to_student(...)` visibility check (Bundle 2)
- Existing auth + grading flow

**Auth middleware update (REQUIRED).** `backend/auth.py:68-76` defines `PUBLIC_PREFIXES` — a list of URL prefixes that bypass the JWT auth check (because they use `X-Student-Token` instead, NOT a Supabase JWT). The existing `/api/student/submit/` is in this list at line 70. The NEW `/api/student/class-submit/` URL is NOT covered by that prefix (`startswith()` matching), so the JWT middleware would reject all class-submit requests with `{"error": "Authentication required"}`, 401 BEFORE the route handler runs.

Add a new entry to `PUBLIC_PREFIXES`:
```python
'/api/student/class-submit/',  # Class-based authenticated submission (X-Student-Token, not JWT)
```

Place it next to the existing `'/api/student/submit/'` entry so the comment block stays grouped.

**Security test update.** `tests/test_security.py` has a test that asserts `is_public_route('/api/student/submit/ABC123') is True`. Add the parallel assertion: `is_public_route('/api/student/class-submit/abc-uuid') is True`.

The join-code path at `backend/routes/student_portal_routes.py::submit_assessment` stays at `/api/student/submit/<code>` — anonymous portal untouched. Its `/api/student/submit/` PUBLIC_PREFIXES entry stays.

### Frontend — one production callsite update

(NOTE: the spec covers ONE production source callsite. Repo-wide search may show other references in e2e tests, load tests, README/docs — most of those are for the join-code path or general docs and don't need updates. Only the StudentPortal.jsx auth-branch fetch at line 216 needs the URL change.)

In `frontend/src/components/StudentPortal.jsx:216` change:

```javascript
// Before:
var response = await fetch("/api/student/submit/" + contentId, { ... });

// After:
var response = await fetch("/api/student/class-submit/" + contentId, { ... });
```

The companion branch at line 227 (`api.submitStudentAssessment(joinCode, ...)`) is the anonymous join-code path. It stays untouched and continues hitting `/api/student/submit/<code>`.

`frontend/src/services/api.js:752` (the `submitStudentAssessment` function) is also unchanged — it serves the join-code path.

### Tests

1. **Re-enable the skipped test** in `tests/test_student_content_visibility.py::test_submit_student_work_404_for_non_targeted`. Remove the `@pytest.mark.skip(...)` decorator. Update the POST URL from `/api/student/submit/ct-1` to `/api/student/class-submit/ct-1`. The test now actually exercises the visibility helper on the auth submit path.

2. **Keep the direct-invoke pattern** in `tests/test_student_account_coverage.py::TestSubmitStudentWork`. The pre-existing comment at lines 207-211 about routing conflict is now stale. Update it to:
   ```python
   """POST /api/student/class-submit/<content_id>

   Phase 4.2 (2026-04-27): URL was previously /api/student/submit/<X>,
   which shadowed student_portal_bp's join-code handler. Migrated to
   /api/student/class-submit/<content_id> to disambiguate.
   The direct-invoke pattern is kept for parity with existing tests;
   client.post(...) would also work now that routing is distinct.
   """
   ```
   The `_call_submit` helper's f-string still works as-is (it builds whatever URL the caller passes).

3. **One new regression test** in `tests/test_student_content_visibility.py` that confirms routing reaches `submit_student_work` (not `submit_assessment`). Asserts on the handler-specific 401 body to distinguish the two failure modes:
   ```python
   def test_class_submit_url_reaches_auth_handler():
       """Phase 4.2 routing fix: POST /api/student/class-submit/<content_id>
       must reach submit_student_work (auth path), NOT submit_assessment
       (join-code path). Without X-Student-Token, _validate_student_session
       returns None and the handler returns:
           jsonify({"error": "Not logged in"}), 401
       The join-code path's failure mode would be:
           jsonify({"error": "Assessment not found"}), 404
       Asserting on the body distinguishes which handler ran.
       """
       from backend.app import app
       with app.test_client() as client:
           resp = client.post('/api/student/class-submit/abc-12345-uuid', json={'answers': {}})
       assert resp.status_code == 401, f"Expected 401, got {resp.status_code}: {resp.get_data(as_text=True)}"
       body = resp.get_json() or {}
       assert body.get('error') == 'Not logged in', \
           f"Expected 'Not logged in' (auth-path body), got {body!r}"
   ```

## Edge cases & invariants

| Case | Expected behavior |
|---|---|
| Anonymous student POSTs to `/api/student/submit/ABC123` (6-char code) | Goes to `submit_assessment`. Unchanged. |
| Class-based student POSTs to `/api/student/class-submit/<UUID>` (new URL) | Goes to `submit_student_work`. Auth + visibility checks fire. Submission succeeds. |
| Class-based student POSTs to old URL `/api/student/submit/<UUID>` (stale frontend) | Goes to `submit_assessment` (still). Returns 404 "Assessment not found". Same behavior as today. After Railway deploys the new frontend, no real client should hit this. |
| Old test `test_submit_student_work_404_for_non_targeted` | Now active, exercises helper, asserts 404 for non-targeted student. |
| Direct-invoke tests in `TestSubmitStudentWork` | Still pass via the `_call_submit` helper. The bypass pattern is preserved for parity. |

## Non-goals

- Don't change the join-code path's URL.
- Don't refactor `StudentPortal.jsx` component structure (just URL change).
- Don't apply visibility helper to `submit_assessment` — `published_assessments` schema has no `target_student_ids` field; targeting is a class-content concept.
- Don't add a generic shadow-detection test or lint rule. YAGNI.
- Don't backport this URL change to old client builds. Stale frontends will continue 404'ing on old URL until they refresh — same effective behavior as today.

## Risk

**Low.** All changes are mechanical:
- 1 line in backend route decorator
- 1 line added to `PUBLIC_PREFIXES` in `backend/auth.py`
- 1 line in `tests/test_security.py` mirroring the existing assertion
- 1 line in frontend fetch URL
- Test updates (skipped → active; one new regression)
- One CLAUDE.md doc line update

The auth submit path was already broken in production, so this can only improve things. The join-code path is untouched, so anonymous submissions are unaffected.
