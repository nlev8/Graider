# Phase 4.1 — Portal Grading → Celery Migration (Gemini Review)

**Date:** 2026-04-16
**Status:** Post-Codex HOLD revision. Seeking Gemini second opinion before writing the spec file.

## TL;DR

Graider's portal submission grading currently runs on unbounded `threading.Thread` spawns inside gunicorn web workers. Work dies when Railway restarts (student submissions stuck as `grading_deferred`). Phase 4.1 of the codebase improvement roadmap migrates the **join-code portal grading path** to a Celery + Redis task queue on a separate Railway worker dyno. Class-based student-account grading stays on threads for now (Phase 4.1b). Codex gave HOLD with 6 concrete concerns which have been incorporated below. Seeking Gemini review before finalizing the spec.

### Pain points driving this work

1. **Durability:** Daemon threads die on Railway restart. Portal submissions get stuck; students lose their work and teachers re-grade manually.
2. **Unbounded concurrency:** Each submission spawns a new thread. Under a classroom-sized load spike, thread count scales linearly with concurrent submissions competing for 2 gunicorn worker slots.
3. **Web/worker coupling:** Grading runs in the same gunicorn worker process that serves HTTP. Long grading blocks request handling.

## Proposed Design

### Stack (fixed by roadmap)

- Celery + Redis (roadmap verbatim)
- Railway worker dyno (new service, same repo)
- Redis reused from existing `REDIS_URL` on DB 1 (DB 0 stays for limiter/session), controlled via explicit `CELERY_BROKER_URL` env var

### Scope tightening (post-Codex)

Phase 4.1 covers **join-code path only** (`/api/student/submit/<code>` in `student_portal_routes.py`). The class-based path (`student_account_routes.py`) is Phase 4.1b. Thread function `run_portal_grading_thread` stays callable until 4.1b migrates the class path.

### Components

| New file | Responsibility |
|---|---|
| `backend/celery_app.py` | Celery instance, broker config, Sentry worker init with `CeleryIntegration` |
| `backend/tasks/grading_tasks.py` | `grade_portal_submission(submission_id, teacher_id, supabase_table, *, district_id, user_id)` task |

| Modified file | Change |
|---|---|
| `Procfile` | Add `worker: celery -A backend.celery_app worker --loglevel=info --concurrency=4` |
| `backend/api_keys.py` | Refactor `get_api_key` to accept explicit `district_id` kwarg (mirrors Phase 3b1 PR4's `_auto_fix_flagged_questions` explicit-context pattern). Flask `g` fallback kept for backward compat of existing callers. |
| `backend/routes/student_portal_routes.py` | Replace `threading.Thread(...)` with `grade_portal_submission.delay(...)` behind `CELERY_PORTAL_GRADING` feature flag. Thread fallback kept while flag is off. |
| `.env.example` | Document `CELERY_BROKER_URL` + `CELERY_PORTAL_GRADING` flag |
| `requirements.txt` | Add `celery` (currently only `redis` is present) |

### Task design

**Signature:**
```python
@celery_app.task(
    name='grading.portal_submission',
    bind=True,
    acks_late=True,
    autoretry_for=(TransientError, TimeoutError),
    retry_backoff=True,
    retry_backoff_max=300,
    max_retries=3,
    time_limit=900,       # 15 min hard
    soft_time_limit=840,  # 14 min soft
)
def grade_portal_submission(
    self,
    submission_id: str,
    teacher_id: str,
    supabase_table: str,  # 'submissions' (join code) or 'student_submissions' (class, future)
    *,
    district_id: str | None = None,
    user_id: str | None = None,
) -> None:
    ...
```

**Body outline:**
1. **Dedup lock:** `SET submission_lock:{submission_id} {worker_id} NX EX 900` via Redis client. If returns nil → another worker already grading, return silently.
2. **Fetch state:** Load the submission row from Supabase. If status != `'queued'` or `'pending'`, return (already processed).
3. **Grade:** Call existing `run_portal_grading_thread` body logic (extracted to a pure function `grade_portal_submission_sync(...)` that takes all dependencies as params — no Flask context).
4. **Persist:** Update submission row (status=graded), append history, release lock.
5. **On exception:** let Celery retry. After max retries, mark submission `failed` with error message; Sentry captures exception via `CeleryIntegration`.

### Durability guarantees

| Concern | Current (threads) | After (Celery) |
|---|---|---|
| Railway restart mid-grade | Submission stuck at `grading_deferred` | `acks_late` → broker redelivers to next available worker |
| Worker crash mid-grade | N/A (single process) | Same — broker redelivers |
| Double-grade from retry | N/A (single thread per submission) | Prevented by SETNX lock + status check |
| Transient OpenAI/Supabase failure | Silent score=0 with log entry | Celery retries with exponential backoff; after 3 fails → `failed` status + Sentry alert |
| Unbounded concurrency | Yes (1 thread per submission) | Bounded by `--concurrency=4` per worker dyno |

### Safety rails

1. **Feature flag** `CELERY_PORTAL_GRADING` — default OFF. Route checks flag; if on, enqueue; if off, spawn thread (old path). Hard fallback for emergency rollback.
2. **Explicit broker URL** — fails fast at worker boot if `CELERY_BROKER_URL` unset. No silent fallback to wrong infra.
3. **Pre-PR0 schema gate** — verify `submissions` + `student_submissions` have the status columns the task reads/writes. If missing, schema migration ships first.
4. **Sentry CeleryIntegration** — task exceptions surface in existing observability. BetterStack uptime monitor already covers the web side.
5. **Deployment order enforcement** — worker dyno must be live before flag flips. Documented as manual operational step for PR3.

### Migration plan (5 PRs, sequential merge required)

**PR0 (conditional)** — Schema migration if submissions/student_submissions lack needed status fields. Might be a no-op if schema is already correct.

**PR1** — Infrastructure only. `celery_app.py` + `tasks/grading_tasks.py` (stub task that logs and returns) + Sentry worker init + Procfile worker entry + `requirements.txt` celery dep + `.env.example` docs. Flag remains disabled. No behavior change in portal grading.

**PR2** — Port the grading body into the task with explicit-context refactor:
- Extract `grade_portal_submission_sync(...)` from `run_portal_grading_thread` body (pure function, no Flask, takes all deps as params)
- `run_portal_grading_thread` becomes a thin wrapper that pulls from Flask `g` and calls the pure function (keeps class-based path working unchanged)
- Task calls the pure function with explicit args
- Refactor `get_api_key` to accept explicit `district_id` kwarg
- Add SETNX dedup in the task
- Feature flag wiring at the route

**PR3** — Enable flag on prod; 48-hour monitoring window. Dashboard/queries to watch: task success rate, median task duration, submission status distribution, Sentry error rate.

**PR4** — Delete portal-routes thread path + portal-specific SIGTERM handler (keep `run_portal_grading_thread` function since class-based path still uses it).

## Questions for Gemini

1. **Architecture:** Is the 5-PR sequence the right decomposition, or should PR0+PR1 merge (infra + schema), or PR3 be removed (since it's operational-only, no code)?

2. **Dedup correctness:** Is Redis `SETNX` with 15-min TTL sufficient for exactly-once semantics, or do we need a more robust idempotency-key strategy (e.g., store `grading_task_id` on the submission row and check/compare)?

3. **Scope split:** Phase 4.1 (join code) vs 4.1b (class-based) — is the split value worth the temporary split-mode state where half the system is on threads and half on Celery? Or should we bite the bullet and migrate both in 4.1?

4. **Celery vs. alternatives:** The roadmap fixed "Celery + Redis" in 4/12. Is there a lighter alternative worth reconsidering (RQ, Dramatiq) given our scale, or is Celery correct here?

5. **`run_portal_grading_thread` refactor:** The plan extracts a pure `grade_portal_submission_sync(...)` with the thread function as a thin wrapper. Is this the right seam, or should we take a different approach (e.g., inline the body into the Celery task and let the class-based path call through a shim that goes through the broker)?

6. **Worker dyno sizing:** `--concurrency=4` per worker with 1 worker to start. Any heuristic from similar migrations you'd recommend for Railway scale?

7. **Deployment order enforcement:** Currently rely on manual operational step ("worker live before flag flips"). Is that acceptable, or should we add a health-check gate (web refuses to set flag if worker heartbeat missing)?

8. **Feature flag mechanism:** We currently read the flag via `os.getenv('CELERY_PORTAL_GRADING')`. Should this be a Supabase-stored config for runtime flip without redeploy, or env-var is fine for an MVP with 1 deployment environment?

9. **Any other hidden issues:** On a fresh read of the current portal grading architecture (`backend/services/portal_grading.py:1-630`), are there dependencies on Flask context or shared state we haven't accounted for?
